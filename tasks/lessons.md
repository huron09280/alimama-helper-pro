# Lessons

## 2026-04-03 用户修正：授权校验应按“使用时触发”，而非空闲期轮询
- 问题：extension 在页面空闲阶段就预检/续租，会把临时网络波动放大成“功能已锁定”干扰。
- 纠偏规则：extension 模式授权应改为 on-demand；仅在用户触发插件交互（插件 UI 点击/键盘）时才发起校验。
- 边界规则：userscript 可保留启动预热；extension 与 userscript 的校验时机需要显式分支，避免互相影响。

## 2026-04-02 用户修正：已授权不等于可通过 policyToken 验签
- 问题：授权状态为 allow 但 extension 仍会锁定，根因可能是签名格式不兼容（DER）而不是授权名单问题。
- 纠偏规则：JWT `ES256` 在本项目必须输出 P-256 原始签名（`ieee-p1363`，64 字节），禁止使用 Node 默认 DER 编码。
- 验证规则：每次改动 policy token 签发链路后，必须做三步验收：`token 签名长度=64`、`WebCrypto subtle.verify=true`、`build/channel claim 与客户端一致`。

## 2026-04-02 用户修正：必须确认“写进云库”，不能只看接口返回
- 问题：`admin/state` 可能看起来正常，但若 FC 部署包缺少 `tablestore` 依赖，运行时仍会在写入/读取阶段报 `ERR_MODULE_NOT_FOUND`，造成“授权操作有记录但云端状态不稳定”的错觉。
- 纠偏规则：云存储问题排查必须三段式闭环：`日志回溯（历史异常） -> 包含依赖重新部署 -> 部署后新请求日志无同类错误`，不能只以一次接口 `200` 判定修复。
- 部署规则：FC 通过 ZIP 更新代码时，涉及 SDK 的功能必须把依赖一并打包（`node_modules` 或可用的层）；发布后必须在新时间点确认不再出现 `failed to import tablestore sdk`。

## 2026-04-02 用户修正：授权状态禁止本地兜底，必须全量上云
- 问题：授权状态在云存储异常时若静默回退到文件/内存，会导致多实例状态漂移，表现为“有时授权、有时失效”。
- 纠偏规则：线上授权状态必须以 Tablestore 为唯一持久化来源；未配置或不可用时应显式返回错误，禁止隐式回退本地存储。
- 守护规则：新增源码契约测试固定“`syncStateBeforeRequest`/`persistStateChanges` 无 `file` 回退 + 云存储异常统一 `503`”，避免后续回退。

## 2026-04-02 用户修正：shopName 兜底不能误抓导航词
- 问题：`shopId` 邻近文本兜底若只按“近行非空”取值，容易误抓“消息中心/工作台”等导航词，导致 `shopName` 由空变错。
- 纠偏规则：店铺名兜底必须采用“`shopId` 锚定 + 标签语义 + 强信号过滤”组合策略；邻近行候选需命中店铺强特征或明确店名标签，且通用导航词必须过滤。
- 验证规则：每次调整 `shopName` 识别链路后，至少在真实阿里妈妈页面做一次 `shopId -> shopName` 结果校验，确认候选非空且非噪音词；若在锁定遮罩态发生回填，必须立即重渲染遮罩，避免 UI 停留旧值 `-`。

## 2026-04-02 用户修正：新店铺默认授权 3 天
- 问题：新店铺首次使用插件时若仍保持“未授权”，会导致运营侧需要手工放行，且体验上表现为“在用但不可用”。
- 纠偏规则：授权服务端在 `verify` 识别到“首次出现且未吊销”的店铺时，自动写入默认有效期授权（默认 3 天，可由 `AM_LICENSE_DEFAULT_AUTH_VALID_DAYS` 配置）。
- 守护规则：新增回归测试固定“自动授权入口、吊销保护、默认天数来源、verify 调用顺序”，防止后续回退。

## 2026-04-01 用户修正：到期日期编辑必须就近放在“授权到期”列
- 问题：把日期编辑放在操作区会导致“到期信息显示”和“到期修改入口”分离，且同页重复出现到期内容，增加操作负担。
- 纠偏规则：时间类字段优先就地编辑，必须贴近对应数据列（本次为授权到期列），避免跨列操作。
- 交互规则：日期修改使用 `change` 直接提交，不再要求额外下拉动作或二次点击执行。

## 2026-04-01 用户修正：线上入口管理页必须与本地管理页同源
- 问题：本地 `dev/license-admin.html` 已有“首次使用/授权到期/日期直设”，但线上访问的是 `index.mjs` 内嵌旧页面，导致用户仍看不到新能力。
- 纠偏规则：同一管理功能若同时存在“本地页 + 线上内置页”，必须强制同源（同一模板文件）或自动同步，禁止双份页面独立演化。
- 守护规则：新增测试同时断言模板字段与服务端加载方式，避免后续再次出现“本地可见、线上缺失”。

## 2026-04-01 用户修正：授权台必须可见“首次使用”并支持到期日期直设
- 问题：仅显示最近活跃不足以判断授权生命周期；没有日期直设时，运营需手工换算 ISO 时间，操作成本高且易错。
- 纠偏规则：授权管理页必须提供两项可见性能力：`firstSeenAt`（首次使用）+ `authExpiresAt`（到期时间）；并提供行级日期控件直接设置到期日。
- 数据规则：首次使用时间应在服务端首次 verify 命中时落库，后续请求不得覆盖为更晚时间。

## 2026-04-01 用户修正：店铺授权需带业务有效期，默认3天
- 问题：仅有 `enabled/revoked` 两态无法覆盖“临时授权”场景，运营无法按周期自动失效。
- 纠偏规则：`admin/allow enabled=true` 必须支持有效期参数，并在未显式指定时默认生效 3 天；`verify` 必须区分 `shop_not_allowed` 与 `shop_license_expired`。
- 交互规则：管理页行级授权操作需直接暴露固定周期（3天/1月/3月/1年/永久）且默认项清晰可见。

## 2026-04-01 用户修正：下拉动作不应再要求二次点击
- 问题：行级操作改为下拉后若仍保留“执行”按钮，会产生不必要的二次确认成本，不符合高频运维场景效率预期。
- 纠偏规则：当用户明确要求“下拉后直接执行”时，必须改为 `change` 触发即时执行，并在执行后重置下拉到默认项。
- 验证规则：浏览器验证需覆盖“选中动作后自动触发请求与日志反馈”，而不只是检查 DOM 存在。

## 2026-04-01 用户修正：高风险行级操作默认用“下拉选择 + 执行”
- 问题：同一行并排放多个高风险按钮（授权、取消授权、吊销、取消吊销）容易误触，尤其在批量排障场景。
- 纠偏规则：对授权管理类行级操作，优先采用“单下拉选择动作 + 明确执行按钮”的两步交互，减少误操作。
- 维护规则：页面交互改造后，必须同步更新静态契约测试断言（选项值、执行入口）与文档操作说明，避免 UI 与测试脱节。

## 2026-04-01 用户修正：shopId 与 shopName 不能假设同源
- 问题：真实页面里常出现 `shopId` 可从 storage/request 识别，但店铺名称只存在于 DOM 文案，若识别链路要求“同源同时命中”就会稳定上报空 `shopName`。
- 纠偏规则：授权识别必须支持“异源拼接”策略：`shopId` 先命中后，允许用低风险店名候选做回填，并结合异步重试补录到服务端。
- 测试规则：针对授权模块的源码契约测试需要覆盖“松耦合店名兜底入口”和“最终 identity 回填接入”两个断言，防止后续回退为空名上报。

## 2026-04-01 用户修正：多店铺场景必须提供“空店名手工回填”兜底
- 问题：某些店铺会稳定上报 `shopId` 但 `shopName` 为空（例如 `114025783`），仅靠自动识别会导致管理页长期显示 `-`，影响运营判断。
- 纠偏规则：授权管理页必须提供行级“设置店名”能力，并通过管理接口把 `shopName` 落库，不依赖插件端立即修复。
- 状态规则：回填店名时必须保持当前 `allowed/revoked` 状态不变，禁止“改名操作”隐式变更授权结果。

## 2026-04-01 用户修正：授权状态持久化优先使用云数据库
- 问题：仅做本地文件/内存持久化，在多实例与扩缩容场景仍可能出现跨实例覆盖或读取时序不一致。
- 纠偏规则：授权核心状态默认优先落云数据库（本项目为 Tablestore），文件持久化仅作为兜底，不作为主方案。
- 运维规则：管理页排障时必须先确认 `admin/state.storage.mode`，确保当前实例运行在预期持久化模式。

## 2026-04-01 用户修正：授权 API 对齐必须优先读取仓库内服务端源码
- 问题：仅依赖控制台页面或记忆做接口对齐，容易把本地管理页契约对到过期/猜测路径（如 `admin/shops`），导致“页面可打开但接口不可用”。
- 纠偏规则：用户给出服务端源码路径时，必须以仓库内实现（本次为 `services/license-server/index.mjs`）作为唯一接口事实来源，再同步前端、文档和测试。
- 校验规则：对齐后至少验证三件事：真实路由、真实鉴权头（`x-am-admin-token/x-admin-token`）、定向测试与浏览器冒烟均通过。

## 2026-04-01 用户修正：授权管理页空列表需区分“无数据”与“被筛选”
- 问题：当 `keyword/status/activeWithinHours` 过滤导致 0 条时，若仍提示“暂无数据”，会误导为服务端没上报在用店铺。
- 纠偏规则：空态必须显示当前筛选条件并提示如何恢复（重置筛选或把 `activeWithinHours` 设为 `0`）；“重置筛选”应使用最宽松默认并触发重新拉取。

## 2026-04-01 用户修正：授权锁定排障必须先对齐“当前页面 shopId”
- 问题：如果只盯着某个报错店铺回放 `allow/verify`，但当前页面实际运行的是另一个 `shopId`，会出现“管理页看起来恢复了，但页面仍被锁”的假修复。
- 纠偏规则：真实页面出现“功能已锁定”时，先读取页面头部 `ID：xxxx` 与遮罩中的 `shopId`，以当前页面店铺为第一优先进行授权回放与验证。
- 校验规则：闭环至少覆盖三项：`/v1/license/verify` 返回 `authorized=true`、`window.__AM_LICENSE_STATE__.authorized=true`、页面不再出现“店铺授权校验失败，功能已锁定”遮罩。

## 2026-04-01 用户修正：admin/allow 空 shopName 不能覆盖已有店铺名
- 问题：管理页行级按钮调用 `admin/allow` 通常只传 `shopId + enabled`，若服务端直接用空 `shopName` 覆盖，会导致店铺名称被清空并长期显示 `-`。
- 纠偏规则：`admin/allow` 的名称更新必须遵循优先级：`payload.shopName > MEMORY_SHOP_STORE.shopName > MEMORY_ACTIVE_SHOP_STORE.shopName`，空值或占位值（`-`/`--`）不得覆盖已有有效名称。
- 回填规则：`verify` 成功拿到有效 `shopName` 后，应回填 `MEMORY_SHOP_STORE`（仅限已允许店铺），保证 `admin/state.shops` 稳定显示店铺名。

## 2026-03-25 用户修正：tooltip/柱子联动要按标签内容，不按索引
- 问题：各周期独立排序时，同一个 `labelIndex` 可能对应不同省份/城市，导致 tooltip 与跨周期柱子对比错位。
- 纠偏规则：悬停联动和对比索引必须使用标签内容键（labelKey）进行匹配，`labelIndex` 仅作渲染位置信息，不能作为跨周期对齐依据。

## 2026-03-25 用户修正：省份/城市排序默认要按各周期，主周期优先改为可切换
- 问题：如果默认直接按 90/30/7/3 稳定排序，会改变用户当前“按各周期看本期排序”的阅读习惯。
- 纠偏规则：省份/城市默认使用各周期独立标签顺序；仅在用户点击排序图标时切换为主周期优先（90→30→7→3），并支持再次点击回退默认。

## 2026-03-25 用户修正：省份/城市稳定排序要以90天为主周期
- 问题：仅按全周期累计值排序时，鼠标对比和阅读顺序不一定体现“90天同期”优先，容易偏离业务主视角。
- 纠偏规则：动态标签维度（省份/城市）在已对齐同标签前提下，稳定排序必须先比较 `90天`，缺失再依次比较 `30/7/3天`，最后才回退累计值和初始顺序。

## 2026-03-25 用户修正：省份/城市跨周期比较必须先做标签对齐
- 问题：若每个周期按自身返回顺序渲染，跨周期对比会退化成“第 N 个对第 N 个”，省份/城市会错位。
- 纠偏规则：对动态标签维度（省份/城市）必须先构造“全周期稳定标签顺序”，各周期按统一标签列表回填缺失值后再比较。

## 2026-03-25 用户修正：明确“只改下拉弹窗，不改触发器”
- 问题：用户说“下拉弹窗样式”时，若同时改触发器，容易偏离需求边界。
- 纠偏规则：UI样式需求出现“弹窗/面板/下拉层”等明确词时，优先限制改动范围到对应层级，避免顺手修改触发器或其他关联控件。

## 2026-03-25 用户修正：下拉视觉需对齐“周期置顶表头”样式
- 问题：自定义下拉虽可用，但视觉与看板内部“周期置顶表头”风格不一致，整体感割裂。
- 纠偏规则：同一看板内新增控件需优先复用既有主视觉语义（白玻璃背景、轻边框、blur、阴影），避免出现独立风格分叉。

## 2026-03-25 用户修正：自定义下拉必须防止继承 nowrap 导致横排
- 问题：头部容器使用 `white-space: nowrap` 时，自定义下拉若不重置容器换行且选项未强制块级，会出现“选项横向一排”。
- 纠偏规则：自定义下拉组件必须显式加两条兜底样式：下拉容器 `white-space: normal`，选项节点 `display: block`。

## 2026-03-25 用户修正：商品下拉必须改为网页自定义组件，不能用系统下拉
- 问题：即使去掉边框，原生 `<select>` 的展开层仍由系统/浏览器接管，样式与站内 UI 无法统一。
- 纠偏规则：涉及“统一网页风格”的下拉交互时，必须使用自定义触发器 + 自定义选项面板（HTML/CSS/JS），禁止继续依赖原生下拉弹层。
- 交互规则：自定义下拉至少支持三件事：点击触发器展开/收起、点击选项触发业务更新、点击外部或 `Esc` 关闭弹层。

## 2026-03-24 用户修正：局部刷新必须先替换旧指标缓存
- 问题：`itemdeal` 局部刷新若直接 merge 成功结果，失败周期会残留旧商品数据，导致切换后“新旧商品混展示”。
- 纠偏规则：局部刷新必须先按 `metric|period` 清空该指标历史结果，再写入本次成功结果；失败周期要显式表现为该指标无数据。

## 2026-03-24 用户修正：刷新中切换不能吞掉后续请求
- 问题：刷新进行中直接 `return` 会导致用户后续切换丢失，UI 选中值与图表数据可能不一致。
- 纠偏规则：刷新中收到新切换时只保留“最新一次”排队请求，当前刷新结束后立即补跑。

## 2026-03-24 用户修正：状态未知商品不能因为“有推广中项”被整体过滤
- 问题：候选过滤若在存在 `active===true` 时丢弃 `active===null`，会误删状态信息缺失但仍可用的商品。
- 纠偏规则：候选过滤只剔除 `active===false`；排序层面再让 `active===true` 优先，兼顾可用性与默认选择稳定性。

## 2026-03-24 用户修正：下拉为空（状态字段误判）
- 问题：不同接口里的 `status` 数值语义不统一，直接按 `0/1` 映射开关会把正常商品误判为暂停，导致下拉被过滤为空。
- 纠偏规则：商品“是否推广中”优先使用 `onlineStatus/isOnline/online` 数值字段；`status/itemStatus` 等仅用于文本关键词判定，禁止直接数值判停。
- 兜底规则：若状态来源不确定，默认 `active = null`（未知）而非 `false`，避免误杀可用商品。

## 2026-03-24 用户修正：商品下拉必须展示标题且过滤暂停，切换只刷新 itemdeal
- 问题：仅展示商品ID会增加识别成本；且把暂停推广商品展示在下拉里会导致“选了不可投商品”。
- 纠偏规则：商品下拉文案必须包含 `商品标题 + 商品ID`，并在候选聚合阶段优先保留 `active === true`，显式暂停(`active === false`)不展示。
- 刷新规则：商品切换事件禁止触发人群看板全量重跑；必须仅刷新 `itemdeal` 并与既有指标结果合并渲染。
- 工程规则：涉及“局部刷新”需求时，先沉淀结果缓存（按 `metric + period`）再增量更新，避免因全量重查导致性能和可用性回退。

## 2026-03-24 用户修正：计划商品下拉必须“全量商品 + 花费降序首项”
