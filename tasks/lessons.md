# Lessons - 2026-05-30

## L121 AI 点睛详情精简不能删掉当前人群解析
- 触发：优化批量 AI 点睛已选需求就近展开时，用户纠正“人群的具体解析怎么删了”。
- 原因：把“详情太多”理解成删除方案计划、热门词和人群画像所有扩展内容，导致当前人群自己的 `crowdProfileList` 解析也被移除；用户要的是详情靠近点击位置并减少远离卡片的大块堆叠，不是丢失人群解释信息。
- 规则：AI 点睛需求详情精简时，必须保留当前 active 人群的具体解析/画像解析，可改为紧凑就近展示；只能移除远离卡片、跨人群或重复的大块信息。验收要同时检查展开位置、关键词完整显示和当前人群解析仍存在。

## L120 批量+ 不重复原生入口，批量改名要做名称批处理
- 触发：用户纠正“原生有的，不需要重复添加了，批量修改名称都不能修改，也没有批量修改名称的操作，例如批量在前面增加，批量在后面增加，批量替换等批量操作名称的动作”。
- 原因：上一轮把原生 `批量计划设置` 中已有的预算、资源位、地域、分时、出价和计划组入口重复加入 `批量+`，误解了“该有的都需要增加”。真正缺口不是复制原生菜单，而是插件自有的计划名称批处理能力。
- 规则：`批量+` 只补原生没有或插件增强明显的能力；原生已有入口不要重复添加。`批量修改计划名称` 必须提供批量前缀、批量后缀、批量替换等名称批处理动作，并保留逐行预览/微调和写请求守卫验证。

## L119 批量+ 功能补齐必须以原生批量修改清单为事实源
- 触发：完成 `批量修改计划名称` 后，用户继续指出“批量修改的功能，该有的都需要增加”，说明只补单个批量项不足以满足 `批量+` 的产品语义。
- 原因：`批量+` 是对原生批量计划设置的增强入口，不能只按当前代码已有 action 或临时需求逐项猜测；否则会漏掉原生菜单中已存在的批量修改项，例如预算、状态、筛选人群等。
- 规则：补齐 `批量+` 能力时，必须先用真实 `one.alimama.com` 页面只读采集原生 `批量计划设置` 菜单和可打开弹窗，以原生菜单项、真实接口和安全边界作为事实源；新增项要优先复用原生接口/弹窗合同，写请求必须用 Chrome MCP 守卫验证，不得真实改计划。

## L118 词包新增合同不能改变源流量智选开关
- 触发：复制 `69514602419 / E7pro_自定义` 后，用户指出人群/关键词相关设置里的 `流量智选` 被打开，但原计划没有打开；上一轮为绕过 `词包出价不能为空` 把流量智选详情态统一归一为开启态。
- 原因：新增接口要求词包对象具备合法出价合同，不代表可以把源计划关闭态改成开启态。把 `wordPackageList` 识别为流量智选后无条件设置 `onlineStatus/status=1`、策略 `onlineStatus=1`，会把复制语义从“保持源状态”变成“强制开启”。
- 规则：复制计划时词包归一化只能清理旧计划/单元 ID 并补齐 `bidPrice` 等新增接口必需字段；顶层 `onlineStatus/status` 和 `strategyList[].onlineStatus` 必须继承源值。源显式关闭时复制后仍关闭；只有 AI 点睛源词包缺失且服务端新增必需时，才允许注入默认开启的流量智选兜底。

## L117 关键词旁边人群复制必须锁定 rightList 与集合一致性
- 触发：复制 `69514602419 / E7pro_自定义` 后，新计划 `81322658190` 的关键词、词包、资源位、全能调价和创意字段都已带上，但只读回读发现源/目标人群同为 23 个、名称集合交集却为 0。
- 原因：关键词自定义计划的人群读取来自按 `campaignId+adgroupId` 的 `/crowd/findList.json`，源数据写入 `adgroup.crowdList`；但最终提交单元还存在 `rightList`，旧逻辑会保留模板或推荐态 `rightList`。服务端落库可能优先按 `rightList` 识别人群，导致数量看似正确但实际复制成另一批推荐人群。
- 规则：关键词旁边人群复制必须把源 `adgroup.crowdList` 同步到 `adgroup.rightList`，并在最终 `/solution/addList.json?bizCode=onebpSearch` 出口再次强制两者一致。验收必须比对源/目标人群 ID 或名称集合，不得只看 `crowdCount`。

## L116 关键词复制的投放资源位是通用高级设置合同
- 触发：用户再次指出 `E7pro_自定义` 复制时 `高级设置 -> 投放资源位` 也没有复制，多次导致计划无法正常投放；此前资源位兜底主要落在 AI 点睛复制路径。
- 原因：关键词推广自定义计划和 AI 点睛计划都依赖 `campaign.adzoneList` 表达资源位开启状态；详情接口缺省或返回空不代表用户关闭资源位。若只在 AI 点睛分支补默认 `淘宝搜索`，自定义计划仍可能以空 `adzoneList` 提交，创建出不可正常投放的计划。
- 规则：关键词当前计划复制必须把 `投放资源位` 当作通用高级设置合同处理。最终 `/solution/addList.json?bizCode=onebpSearch` 请求发出前，若源计划没有显式资源位列表，应补官方默认 `{adzoneId:"114790550288",status:"start"}`；若源计划显式带出资源位列表，则原样保留。验收必须核对最终请求体和新计划高级设置，不得只看 AI 点睛或基础投放字段。

## L115 关键词自定义复制必须把单元级合同作为验收核心
- 触发：用户指出 `E7pro_自定义` 复制时不只关键词缺失，关键词旁边的人群、创意和“全能调价”设置也需要复制；真实复制又暴露流量智选词包按详情态提交会返回 `词包出价不能为空`。
- 原因：关键词推广自定义计划的关键配置大量挂在 adgroup 侧或独立列表接口里，`campaign/get.json` 和 `adgroup/get.json` 不一定直接带出可提交合同；详情态词包/关键词/人群字段还可能和新增态提交合同不同。如果只复制计划级字段或直接透传详情态字段，会创建出无单元、无关键词或缺少出价合同的半成品。
- 规则：关键词推广自定义复制必须显式读取并验证 `/bidword/findList.json`、`/wordpackage/findList.json`、按 `campaignId+adgroupId` 的 `/crowd/findList.json` 和完整 `adgroup/get.json`；最终 `/solution/addList.json?bizCode=onebpSearch` 请求必须核对 `wordList`、`wordPackageList` 顶层与策略出价、`adgroup.crowdList`、`adgroupOcpc` 和创意字段。真实成功标准是响应无 `errorDetails` 且新计划有单元、关键词、词包、人群和单元设置，不能只看 campaign 创建成功。

## L113 复制计划必须覆盖“高级设置”字段而不只看基础投放字段
- 触发：AI点睛复制计划前几轮只重点核对了 `sourceChannel/searchUpgradePxb/状态/预算/出价/AI点睛/商品`，用户继续指出真正原因是“高级设置里的没有复制过来”。
- 原因：关键词推广的可投放效果不只由基础字段决定，高级设置还可能包含地域、分时、资源位溢价、屏蔽词、优质计划防停投、创意/资源位/人群等隐藏配置；如果只看 `campaign/get.json` 的顶层字段或复制弹窗摘要，会漏掉高级设置内部差异。
- 规则：创建/复制类对比必须单独列出高级设置字段清单，并用真实页面或服务端详情证明每个高级设置字段是否保留。修复时应把高级设置作为一个合同集合进入复制白名单和最终 payload 白名单，验收必须覆盖浏览器提交后的服务端详情，而不是只覆盖弹窗或基础字段。

## L114 资源位默认开启不能按 adzoneList null 判空
- 触发：排查源计划 `81211576775` 时只看 `campaign/get.json` 的 `campaign.adzoneList=null`，误判资源位为空；用户截图显示高级设置里 `淘宝搜索` 明确为开启。
- 原因：关键词推广资源位默认状态可能在页面高级设置/提交 payload 中表达为默认开启，详情接口顶层 `adzoneList=null` 不等于用户关闭或无资源位；官方 `addList` 样本中默认淘宝搜索会以 `adzoneId=114790550288,status=start` 表达。
- 规则：核对关键词推广资源位时必须同时看页面高级设置、原生提交 payload 和服务端详情；若详情字段为空但页面显示默认开启，复制组包要保留默认开启语义，不能把空数组当成关闭状态提交。

## L112 写请求守卫验收后必须恢复并刷新确认
- 触发：为验证复制计划点击 `确认生成` 的提交路径，在真实 `one.alimama.com` 页面安装了写请求守卫；验收结束后守卫残留，用户再次点击时看到 `CODEX_GUARDED_WRITE_BLOCKED`。
- 原因：写请求守卫是测试工具，不是产品逻辑；如果只记录守卫拦截结果而不执行 `restore()`、关闭弹窗、刷新页面并确认 `window.__codexCopySubmitGuard` 不存在，就会把测试拦截状态留在用户真实页面。
- 规则：任何 Chrome MCP 写请求守卫验收结束前，必须恢复原始 `fetch/XMLHttpRequest`、删除守卫全局变量、关闭或刷新受影响页面，并用页面脚本确认 `hasGuard:false`、`hasGuardText:false`、业务弹窗无残留；最终回复必须明确守卫已清理。

## L111 复制验收必须覆盖确认生成后的提交路径
- 触发：复制计划一览窗打开后不再显示 `bizCode未找到`，但用户修正真正报错发生在点击 `确认生成` 后；上一轮只验证弹窗 ready，验收不足。
- 原因：复制链路在确认按钮之后还会重新经过 `submitPreparedCopyCurrentPlan -> copyCurrentPlanByScene -> buildCurrentPlanCopyRequestByScene -> requestOne`，弹窗上下文正确不代表最终 request/meta/runtime 和 URL 合同都正确。
- 规则：创建/复制类缺陷只要用户授权验证确认路径，必须在写请求守卫下真实点击 `确认生成`，记录是否到达写接口、完整 URL、脱敏 payload 关键字段和页面状态；不能用“确认后才会提交创建请求”的 ready 文案替代提交路径验收。

## L110 弹窗确认上下文不能丢失 URL 可推导的 bizCode
- 触发：关键词推广复制计划一览弹窗中，当前 URL 已是 `#!/manage/search`，但确认区显示 `bizCode未找到`。
- 原因：复制链路跨快速上下文、异步详情补齐、弹窗编辑行和提交 options，多层传递中任何一层丢失 `bizCode` 都会让确认校验失败；但页面路由本身已经能确定业务线。
- 规则：复制/批量弹窗的提交上下文必须在最终确认前从 `context.bizCode`、`source.bizCode`、源 campaign、当前 URL 路由和默认业务线依次兜底；当 URL 明确为关键词推广时必须解析为 `onebpSearch`，不得报 `bizCode未找到`。

## L109 创建/复制对比必须核对完整 URL 查询参数
- 触发：手动 AI点睛计划提交使用 `/solution/addList.json?bizCode=onebpSearch`，复制计划提交记录为 `/solution/addList.json`，用户纠正“接口需要用回一样的网址提交”。
- 原因：只对比接口 path 和 payload 会漏掉 query 参数合同；服务端或前端网关可能通过 `bizCode` 查询参数识别业务线，即使 payload 中有类似字段，也不能假设两者等价。
- 规则：真实创建/复制链路对比时，写接口必须记录和对齐完整 URL（path + query）。关键词推广复制仍禁止走官方 `/solution/copy.json`，但受控 `/solution/addList.json` 必须按官方手动链路带上 `?bizCode=onebpSearch`。

## L108 商品 ID 纠偏必须立即改任务规格并保留官方校验证据

## L106 DMP 默认对比人群必须在渠道行为全集加载后应用
- 触发：用户要求 DMP 人群看板“对比人群”默认改为 `全部渠道-成交`。
- 原因：DMP 页面初始运行态里对比人群可能仍是官方当前值（如 `关键词推广-曝光`），如果在可选渠道/行为列表加载前只改按钮文案，后续真实取数或刷新会被页面原始值覆盖；但用户手动选择后又不能被默认值反复覆盖。
- 规则：DMP 默认对比人群应在 `api_goods_insight_portrait_channel_get` 的渠道/行为全集读取完成后，按 `全部渠道 + 成交` 或 `全部渠道-成交` 匹配并写入 `channelBehaviorListCompare`；手动选择对比人群后必须设置保护标记，后续刷新不得再用默认值覆盖。

## L105 人群看板 tooltip 修复必须做文字矩形重叠审计
- 触发：用户反馈鼠标移动到柱子后的弹窗文字有些叠加在一起。
- 原因：tooltip 从纯文本对齐扩展到多指标对比后，长人群名称、占比列、数值列和差异列在真实浏览器里会受字体、列宽和弹窗最大宽度影响；只看源码或单一截图无法证明文字没有互相压住。
- 规则：人群看板 tooltip 应使用受控网格列宽、`min-width:0`、必要换行和等宽数字；完成验收必须用 Chrome MCP 触发真实 hover 后读取 tooltip 内部文本节点 `getBoundingClientRect()`，记录 `overlapCount:0`，不能只凭视觉主观判断。

## L104 DMP 多选画像标签不要强行归一到 100%
- 触发：用户指出 DMP 看板 `特征兴趣` 行占比合计不是 100%，并要求核查是否有问题。
- 原因：`特征兴趣` 官方标签响应为 `tagType:"CHECKBOX"`，同一用户可同时命中多个兴趣；真实 `api_2/analysis/tag/125020` 返回 38 个兴趣项，`rate` 合计约 `747.06%`，这是覆盖率语义而不是互斥分布错误。
- 规则：DMP 自定义画像标签中的多选标签应保留官方 `rate` 作为覆盖率，不得为了凑 100% 二次归一化；需要在 UI 行头或 tooltip 标明“多选/覆盖率”，并用真实接口响应作为验收依据。

## L103 DMP 看板 UI 修正必须按真实控件对齐验收
- 触发：DMP 商品洞察页人群看板继续修正，用户要求下拉浮层左对齐“分析人群/对比人群”按钮、商品 ID 靠左贴近商品名称、0% 不显示、标题改为“达摩盘单品分析”，并补充入口按钮高度和圆角要对齐“切换分析单品”。

- 规则：DMP 看板下拉定位必须以整颗分析/对比按钮为锚点并用真实页 rect 证明左边对齐；商品名称与商品 ID 在 DMP 模式下应组成左侧紧凑上下文块；可见柱状占比为 0 时不渲染“0%”；入口按钮高度和圆角必须从原生“切换分析单品”的可交互控件读取或匹配，不能从内部文字节点取样式。

## L102 DMP 下拉必须挂顶层并做命中验证
- 触发：DMP 人群看板按钮内下拉虽然能通过 DOM/脚本选择，但用户反馈“下拉的窗口没有置顶，被其它挡住无法选择”。

## L100 DMP 人群选择要保留看板系列按钮形态
- 触发：把 DMP 人群看板的分析/对比人群改成原生级联 `select` 后，用户修正“保留‘分析/对比人群’渲染成隐藏/显示按钮，在按钮里做下拉的选择，用设计的下拉选择”。

- 规则：DMP 人群属性选择必须保留分析/对比系列按钮的外观、颜色、显隐切换和 `aria-pressed` 语义；渠道/行为选择用定制下拉面板附着在按钮触发区，不能用裸原生 `select` 替代整组按钮。

## L99 DMP 人群看板行维度必须取自自定义画像标签
- 触发：为 DMP 商品洞察页复用人群对比看板时，先按首屏可见画像维度规划行类别，用户修正“自定义画像标签里有可以分析的人群，按这里来做每一行的类别”。

## L98 人群看板取数缺陷不能用模拟缓存验收
- 触发：修复人群对比看板过去 7 天省份/城市缺失时，新增了基于 `crowdMatrixResultMap` 的模拟缓存行为测试，用户纠正“不要模拟缓存”。
- 原因：这类缺陷发生在真实页面请求、异步补齐、结果回写和渲染链路中；模拟缓存只能证明局部 helper 可写入 Map，不能证明真实查数接口、运行态加载顺序和 UI 展示正常。
- 规则：人群看板/万能查数的真实取数缺陷可以保留静态合同与纯函数测试，但完成验收必须以真实页面重新加载/查询结果为准；不得把模拟 `crowdMatrixResultMap`、mock API、构造数据或缓存注入结果作为“浏览器测试正常”的证据。

## L97 用户指定 Chrome MCP 时不得用 CDP/其它浏览器替代
- 触发：继续优化 Chrome 内存任务中，用户修正“只用chrome mcp”。
- 原因：原生 CDP、Browser 插件或 shell 直连调试端口虽然能做只读探针，但不满足用户指定的验收工具边界，也容易让任务记录把替代验证误写成真实 Chrome MCP 验证。
- 规则：用户明确要求使用 Chrome MCP 时，浏览器验收只能调用 `mcp__chrome_devtools.*` 工具；若 MCP 报 `Could not find DevToolsActivePort`、Transport closed 或连接失败，只能按 Chrome DevTools Ready 流程恢复后重试 MCP。恢复后仍不可用时，记录为 Chrome MCP 阻塞，不得改用原生 CDP、Browser 插件、macOS open 结果或手工探针冒充验收。

## L96 Chrome 内存优化任务必须按项中文提交并对比
- 触发：用户要求“优化现在插件的chrome浏览器占用浏览器内存，预估优化几轮可以优化到最佳，帮我找到最佳优化方案，每优化完成一项或者一次对话，先中文commit再继续”。
- 原因：性能优化容易跨多轮混杂调研、实验、代码和验证；如果不按项提交，会增加复审和回滚成本，也难以判断每一项优化的独立收益。
- 规则：Chrome 内存优化类任务必须把测量对象明确写成 Chrome；每完成一项实质优化都要记录基线、改动、复测结果、前后差值和结论；每完成一项实质优化或本轮对话收口前，先做 diff 自审和验证记录，再用中文提交信息 commit。不得用没有实质优化的空提交替代进展。

## L95 预算破限写入必须先解析服务端硬下限
- 触发：P80 计划预算从 `200元` 降到 `199元` 时，前端提示建议阈值 `2470元`，但真实提交后服务端返回 `日预算不能低于 220.80元`，后续同计划实时硬下限又变为 `223.52元`；用户纠正“先获取服务器返回的最低预算值，再提交”，并补充“设置的预算低于服务器返回的预算值，才提交服务器返回的预算值”。
- 原因：前端/算法建议阈值和服务端硬下限不是同一层限制。若跳过服务端探测直接提交猜测值，容易反复失败；若只记录前端阈值，会把可绕过的建议限制误判为真实最低预算。
- 规则：预算类真实写入必须先用低值试探或已有失败响应解析服务端返回的最低预算值；只有原请求预算低于服务端最低值时，才按服务端最低值重提，原请求已不低于最低值时不得替换重提。验证记录必须同时写明前端建议阈值、服务端硬下限、原请求值、是否触发重提、最终提交值和服务端 `errorCount/list.newDayBudget`。

## L94 设计稿新增浮层后必须做矩形重叠审计
- 触发：插件窗口新版设计稿继续优化 10 轮后，用户指出 HTML 设计稿里出现重叠。
- 原因：视觉上为了表达真实注入态加入了悬浮球、快捷入口、主面板、组建计划、算法护航和下载捕获等多层浮层，但只做静态内容审计会漏掉不同断点下的矩形压叠，尤其是前景窗口压住原生 toolbar、快捷入口压住悬浮球这类问题。
- 规则：静态 HTML/UI 设计稿只要包含绝对定位、浮层、弹窗、toolbar 或响应式断点，完成前必须用浏览器读取 `getBoundingClientRect()` 做关键元素矩形重叠审计；至少覆盖宽屏、常规桌面、临界桌面、窄屏和手机宽度，并记录是否存在页面级横向溢出。

## L93 新版设计稿不能只是干净重排
- 触发：基于真实旧版原稿生成新版代码设计稿后，用户反馈“没有原来的好”，要求优化到明显优于原稿并达到大师 UI 级别。
- 原因：上一版只把旧窗口做成更干净的浅玻璃重排，但缺少真实运行上下文、空间层级、主次节奏、细节材质和控件完成度；对于插件窗口 redesign，用户要的是能指导落地的高质量视觉方案，而不是低风险组件目录。
- 规则：插件窗口新版设计稿必须先保留真实业务语义和信息密度，再在空间秩序、窗口 z 层、控件节奏、状态表达、数据可读性和材质细节上明显超过原稿；静态稿要像真实后台工作台可落地，而不是把控件摆成展示卡片。

## L92 参考动画不能覆盖行内原生按钮形态
- 触发：用户指定复制 `mx_807` 的样式和动画后，又修正“图标与字体的颜色与其它的一样；+ 的图标修改成 12px；圆角大小与左边的按钮一致”。
- 原因：`mx_807` 的 8px 卡片圆角属于独立入口外壳，迁移到计划行操作区时覆盖了同组原生按钮的 `500px` 胶囊圆角；同时 `+` 图标被全局按钮 SVG 规则放大到 14px，视觉层级不一致。
- 规则：行内原生操作组里可以借用参考入口的背景光影或动画，但按钮圆角、文字/图标颜色、字号、字重和小图标尺寸必须以同组原生按钮的 computed style 为准；若参考样式与原生组冲突，原生组优先。

## L91 行内按钮参考图要服从同组原生层级
- 触发：按参考图给计划行“复制”按钮加了蓝色圆形图标底和更强阴影后，用户要求“去掉复制图标下面的深色背景，保持简约；字体的大小与颜色与左边的一致”。
- 原因：参考图是独立大胶囊视觉，直接压缩到计划行操作区会让插件按钮比左侧原生操作按钮更抢眼，破坏后台列表的低噪声层级。
- 规则：计划行、表格行、原生操作组里的插件按钮，除非用户明确要高强调入口，否则图标和文字优先继承同组原生二级按钮的字号、色值和字重；参考图只取结构方向，不能照搬高饱和图标底、强阴影或独立品牌感。

## L90 用户要求原稿一模一样时必须采集真实 DOM/CSS
- 触发：用户要求“先把原来的设计做成现在这个 html”，随后纠正“原稿需要与插件一模一样”。
- 原因：按源码风格手写复刻只能做设计参考，无法满足原稿对照；窗口的真实尺寸、栈层、computed style、按钮状态和运行态内容可能与手写稿有偏差。
- 规则：当用户说“原稿/原来的设计/一模一样/1:1/对比原版”时，必须优先从真实运行态采集 DOM、computed style 和官方窗口宿主；无法安全触发的窗口要标注缺口，不能用近似手画替代。若用户要求“用代码/不用截图”，必须生成可检查的 HTML/CSS/DOM 原稿，不得用截图拼版冒充 1:1。

## L89 创建/复制缺陷复测失败后必须真实闭环
- 触发：复制计划 `E7Pro_AI点睛_测试` 的流量智选词包校验问题上一轮只用构造 dry-run 和只读 dry-run 验证，用户复测仍报同样错误，并明确要求“在浏览器里复制成功为止”。
- 原因：dry-run 只能证明本地组包看起来合理，不能证明实际确认窗、运行态扩展版本、真实提交 payload、服务端校验与创建后状态都成立；尤其复制链路经过弹窗覆盖、运行态桥接和服务端字段校验，任何一层都可能和构造输入不同。
- 规则：涉及创建/复制类缺陷，用户授权真实复制后，完成标准必须是“真实页面确认提交 -> 捕获写接口成功响应 -> 获得新 campaignId/计划名 -> 只读查询或列表搜索确认存在/目标状态正确”；不得再用 dry-run、payload 检查或未提交的浏览器操作替代完成。

## L88 域名禁用要覆盖跳转落地页
- 触发：用户要求 `mai.taobao.com` 不启动捕获报表，真实打开后跳转到 `myseller.taobao.com/home.htm/QnworkbenchHome/`，用户进一步确认该落地页也不要启动。
- 原因：只按用户说的入口域名做守卫会漏掉业务实际运行域名；浏览器跳转后插件真正注入和初始化发生在落地域。
- 规则：处理“某域名/页面不要启动某模块”时，必须用真实浏览器确认最终 `location.href/hostname`，并把入口域名、跳转落地页和子域一并纳入禁用边界；若禁用模块曾隐式初始化共享基础设施，要保留基础设施，避免误伤其它模块。

## L87 下拉弹层修复必须验证命中测试而不只看坐标
- 触发：人群看板商品 ID 弹层上一版真实页 computed rect 已贴近触发器，但用户反馈点击后仍无法选择，被其它层遮住。
- 原因：`getBoundingClientRect()` 和 JS `option.click()` 只能证明 DOM 坐标和程序点击可用，不能证明鼠标命中链路未被 iframe、玻璃层或 stacking context 拦截；fixed 弹层留在复杂弹窗内部时，仍可能被祖先层级或指针事件遮挡。
- 规则：修复跨层弹层时，必须把候选层挂到真正顶层宿主（如 `document.body` 或明确 portal root），并用 `document.elementFromPoint(optionCenter)`、Chrome DevTools 真实点击、`aria-expanded/display/aria-hidden` 变化共同证明可操作；不得只用坐标或 JS 直接调用 click 作为验收。

## L86 关键词推广复制不能走官方 copy
- 触发：用户反馈关键词推广里复制 `E7pro_自定义` 后，发货关键词、地区等隐藏配置没有复制。
- 原因：dry-run 只能证明插件可生成官方复制 payload，不能证明服务端支持；真实 `onebpSearch` 调用 `/solution/copy.json` 返回“该场景暂无此功能”。关键词推广复制必须走受控 `/solution/addList.json`，但旧 addList 组包会重建关键词 bundle，并经过复制白名单、关键词自定义 `prune/normalize` 多轮裁剪，导致关键词、地域、创意、时间折扣、全能调价等源字段有丢失风险。
- 规则：`onebpSearch` 当前计划复制禁止进入官方 `/campaign/copy/campaignCheck.json -> /solution/copy.json` 分支；必须走受控 addList，并把源 `wordList/wordPackageList/crowdList/launchAreaStrList/launchPeriodList/creative*/ruleCommand/enableRuleAuto` 等合同字段纳入白名单和提交 payload。涉及官方复制能力判断时，真实服务端提交结果优先于 dry-run。

## L85 计划列表局部刷新优先用列表 VFrame render
- 触发：排查原生“批量修改每日预算”提交后的刷新方式时，发现页面只是刷新计划列表，而插件 `批量+` 成功路径仍用整页 `window.location.reload()`。
- 原因：真实页面中 `campaign-list` VFrame 的 `render()` 会重新请求 `/campaign/horizontal/findPage.json` 并带上当前筛选、排序、分页参数；`asyncRenderData()` 主要刷新报表和动态模块，不一定重新拉取计划列表行。
- 规则：批量操作成功后需要刷新计划列表时，优先定位当前业务线的 `onebp/views/pages/manage/{search|display|onesite|hky}/campaign-list` VFrame 并调用 `render()`；`asyncRenderData()` 和搜索框回车只能作为兜底，禁止默认整页 reload。

## L84 原生批量预算真实接口以运行态抓包为准
- 触发：真实提交 1 个暂停计划的“批量修改每日预算”时，提交前按旧经验预估允许 `/campaign/updatePart.json`，但实际官方接口是 `/campaign/budget/batchUpdate.json`。
- 原因：同属预算/计划修改的官方入口可能使用不同端点；只按相邻功能或旧验证经验布置写请求守卫，容易漏记真实提交接口，甚至在未授权场景误拦截正确官方请求。
- 规则：涉及真实提交前，先用只读打开官方弹窗和抓包确认目标提交接口；守卫策略应记录所有写请求，只阻断明确不属于本次动作的创建、删除、复制、保存、商品/方案写入口。提交后必须记录请求体中的计划 ID、新预算、响应 `ok/errorCount` 和页面最终状态。

## L83 原生功能关闭插件后生效时优先查全局补丁副作用
- 触发：用户反馈原生“批量计划设置 -> 批量修改每日预算”提交后不生效，但关闭插件后生效。
- 原因：问题不一定来自同屏新增入口（如 `批量+`）的点击拦截，也可能来自插件开启后的全局页面补丁、Magix 组件 monkey patch、请求 hook 或状态同步副作用。
- 规则：遇到“关插件恢复”的原生功能缺陷，先把排查范围扩到所有全局补丁；涉及预算、出价、校验类原生弹窗时，特别检查是否跳过了官方组件 `check()/validate()` 的内部取值同步，修复优先保留官方校验副作用，只在明确的前端限制错误上放行。

## L82 重复强调“窗口外/弹窗外”时先复核运行态 overlay
- 触发：用户再次强调“弹窗‘组建计划’窗口外的增加，白色玻璃渐变，类似组建计划里的增加商品的弹窗外的背景”，而当前顶部任务已切到移除类按钮图标化。
- 原因：连续 UI 迁移中容易把上一切片继续推进，忽略用户最新纠偏其实是在要求当前弹窗外层遮罩聚焦效果。
- 规则：用户重复强调“窗口外/弹窗外/点击后弹窗外”时，先暂停当前非同对象切片，复核运行态 overlay/mask 的 computed style、截图和未触发写请求；只有确认目标层级正确后再继续其它迁移。

# Lessons - 2026-04-15

## L71 组建计划主面板背景不要过度白化
- 触发：按“组建计划背景”纠偏时把主面板提升到 `rgba(255,255,255,0.96) -> rgba(255,255,255,0.88)` 的强白玻璃，用户反馈“面板的背景回退刚刚那种，我不喜欢现在这种”。
- 原因：用户要去掉整屏灰底和内部灰块，但主面板本体已经接受的是更轻、更透明的玻璃面；继续加白会让工作台变成厚白卡片，偏离上一版视觉。
- 规则：组建计划主面板背景默认保持上一版轻透明玻璃 `rgba(255,255,255,0.6) -> rgba(255,255,255,0.2)`；除非用户明确要求“更白/更实”，不要提升主面板和矩阵主背景 opacity。白色玻璃只用于复制计划、添加商品、详情背板等已被明确指定的弹层。

## L70 相邻页面名纠正后必须立即回到用户指定对象
- 触发：UI 迁移中把用户说的“批量计划，也改成这个白色玻璃背景”误联想到 `批量+` 确认窗，用户随后纠正“组建计划的背景才对”。
- 原因：本项目同时存在“组建计划”“批量+”“复制计划”等相邻入口，若只按词面联想而不结合上一句上下文，容易把视觉背景改到错误页面。
- 规则：用户纠正对象后，立即停止旧方向，回到最新指定页面；涉及“批量计划/组建计划/批量+”这类相邻名称时，先用运行态 DOM 和页面入口确认目标，再改源码。

## L69 修改类官方弹窗必须验到确认后的服务端状态
- 触发：线索推广 `批量修改屏蔽人群` 上一轮只验证了能打开官方 `编辑过滤人群` 弹窗并取消，用户继续指出点击确定后再进计划详情看不到成功。
- 原因：打开弹窗只能证明入口链路正确，不能证明 `enterCallback`、提交 payload、差异删除/新增和服务端落库都正确；修改类功能如果不做确认后的详情页或只读接口核对，容易漏掉真实业务失败。
- 规则：涉及配置修改、创建、复制、投放状态等写操作时，在用户授权后必须完成“真实确认提交 -> 读取接口或官方详情页验证最终状态”的闭环；未验到服务端状态前，不得宣称功能已修好。

## L68 Extension 已处理业务失败不要写入 Chrome 错误集合
- 触发：`chrome://extensions` 错误页积累大量 `[EscortAPI] 请求失败`、`CODEX_BLOCKED_CREATE_REQUEST` 和受保护拦截日志，掩盖真正的 runtime 异常。
- 原因：业务/API 失败已经通过 Promise reject、UI 提示或调用方错误处理显式暴露，但同时使用 `console.error` / `console.warn` 会被 Chrome 当成扩展 runtime error/warning 收集。
- 规则：extension 运行态中，已处理且会返回/抛给调用方的控制流失败应使用普通日志级别记录；只有未恢复的启动失败、代码异常或安全边界违规才使用 `console.error` / `console.warn` 进入 Chrome 扩展错误页。

## L67 Extension 内部桥不可依赖本机 debug 全局开关
- 触发：用户在另一台电脑新安装后点击复制计划，报 `计划复制 API 未就绪`。
- 原因：主助手与 `KeywordPlanApi` 分处不同 IIFE；extension 为安全不默认暴露 `window.__AM_WXT_PLAN_API__`，旧复制入口却依赖完整 API 可见，本机有 debug 开关时问题被掩盖，新设备无开关时只剩 openWizard 窄桥。
- 规则：extension 运行态中，用户可点击的创建/复制能力必须走默认安装的内部受控 bridge host 或显式等待 API 就绪；`window.__AM_WXT_KEYWORD_API__` / `window.__AM_WXT_PLAN_API__` 这类 page 全局完整客户端只能作为 debug 面，不能作为生产入口依赖。

## L66 复制成功确认优先页内搜索
- 触发：用户要求复制成功后点击确认不要刷新网页，而是在计划名称搜索框里搜索去掉最右 `_数字` 的计划名。
- 原因：成功弹窗确认使用 URL 跳转或 reload 会打断当前页面状态，也容易让用户误以为还在重新加载整页；复制后的验证只需要在当前列表页按公共计划名筛选。
- 规则：复制/创建成功后的“查看结果”交互优先复用当前页筛选控件并触发原生搜索；搜索词应先归一化新计划名的复制序号后缀，只有找不到页内控件时才做无整页刷新的保守兜底。

## L61 无源字段不能在复制确认窗里新增覆盖
- 触发：用户要求当前计划里没有出价时，复制计划一览窗里的出价编辑不能编辑。
- 原因：复制链路的确认窗是核对和微调源计划已有字段，不应把源计划不存在的字段变成用户可新增覆盖项，否则可能改变计划的原生出价合同。
- 规则：复制确认窗对行级覆盖字段要区分“源计划已有可编辑值”和“源计划不存在值”；不存在的字段应禁用输入、批量编辑也应跳过或禁用，确认提交时不生成对应覆盖字段。

## L60 智能出价展示要带目标明细
- 触发：复制计划一览窗只显示 `智能出价`，用户要求能看出该智能出价具体是促点击、促成交还是促收藏加购。
- 原因：确认窗只展示了出价模式层级，遗漏了源计划 `bidTargetV2/optimizeTarget/subOptimizeTarget/constraintType` 里的目标合同，用户无法在提交前核对复制目标。
- 规则：复制/创建确认窗展示智能出价时，应把只读模式和目标明细组合展示，如 `智能出价（促点击）`；明细只用于核对，不作为可编辑覆盖字段提交。

## L59 官方最终弹窗不能依赖用户可见中间菜单
- 触发：`批量人群设置` 通过官方 `批量计划设置 -> 批量编辑人群` 两段式触发，用户看到弹窗弹出两次。
- 原因：复用官方能力时把原生菜单交互当成实现入口，导致扩展菜单点击后又出现官方中间菜单/弹层，增加一次不必要交互。
- 规则：当扩展菜单项的目标是打开某个最终官方弹窗/抽屉时，优先抓取并直接调用最终弹窗的 `mxModal`/VFrame 参数；只有调试兜底可以定位中间菜单，交付路径不得让用户看到中间菜单。

## L58 多行复制编辑要提供批量填充
- 触发：复制计划一览窗支持逐行编辑后，用户需要按首尾价格生成出价梯度，并一次性设置所有计划预算。
- 原因：复制多计划时逐行填写出价和预算效率低，且等差出价容易人工填错。
- 规则：多行创建确认窗里的重复数字字段要优先提供批量填充能力；梯度类字段应按行数从首值到尾值等差计算，统一类字段应一键写入所有行，并在确认前完成可见回写。

## L57 多列确认窗列宽要内容优先
- 触发：复制计划一览窗加宽后，各列仍按固定百分比分配，用户希望每列按文字内容适应，不要显得过宽。
- 原因：固定百分比会把短文本列（如出价方式、出价价格）拉得过宽，降低一览窗的信息密度。
- 规则：多列确认窗里短文本/数字/枚举列用内容优先的紧凑宽度，长文本列占剩余空间；仍必须验证无横向滚动条。

## L56 确认窗编辑控件优先用轻量下划线形态
- 触发：复制计划一览窗虽然改回原弹窗格式，但编辑框仍是完整边框，用户希望改成只有底部一条横线，视觉更简洁。
- 原因：多列确认窗里每个输入都加完整边框，会让表格显得拥挤，和“提交前快速核对并微调”的轻量场景不匹配。
- 规则：确认窗里的行内编辑控件优先使用无外框、底部线条和轻量 focus 态；只有需要强分组或错误提示时才增加完整边框。

## L55 列表展示值必须限定当前行提取
- 触发：复制计划一览窗出价方式仍显示成 `智能出价`，但用户看到的源计划实际是 `手动出价`；同时一览窗宽度不足出现横向滚动条。
- 原因：出价方式取值优先信任详情接口/内部字段，且 DOM 文案提取没有严格限制在当前计划行，容易扫到同表其它计划的出价文案。
- 规则：列表行操作的确认窗展示值必须优先取触发按钮所在当前行的可见文案，并限制提取范围，不能扫整张表；多列确认窗默认给足宽度并禁止横向滚动条，避免让用户横拉确认关键字段。

## L54 展示字段不要误做成编辑覆盖字段
- 触发：复制计划一览窗把源计划出价方式显示成英文 `smart_bid`，并做成了可编辑字段，用户要求改回原计划显示文案且不能编辑。
- 原因：实现一览窗时把“供用户确认的信息”和“允许用户覆盖提交的字段”合并处理，导致只读信息被当作 API 覆盖项。
- 规则：提交前确认窗里的字段要先区分“可编辑覆盖项”和“只读展示项”；只读项必须优先使用原页面/源计划展示文案，不能用内部枚举值替代，也不能随确认提交写回覆盖。

## L53 用户要求同一个官方窗口时必须复用官方弹窗宿主
- 触发：用户指出 `批量修改屏蔽人群` 不是只调用官方接口或自定义确认框，而是要调用计划详情 `人群 -> 编辑过滤人群` 打开的同一个窗口，并通过该窗口提交批量修改。
- 原因：上一版虽然已抓官方接口，但交互仍是自定义弹窗，和用户要求的官方编辑体验不一致，也无法复用官方组件的选择、校验和数据回填逻辑。
- 规则：用户明确说“同一个窗口/一样的窗口/官方弹窗”时，先定位页面 Magix/VFrame/modal 宿主和官方 view，再复用官方弹窗组件；接口提交只能作为弹窗提交回调的一部分，不能用自定义弹窗替代官方交互。

## L52 官方隐藏弹窗能力不能用详情页跳转替代
- 触发：用户指出人群推广 `批量修改屏蔽人群` 与 `批量人群设置` 不要跳转详情页，官方能力隐藏在计划内 `人群 -> 编辑过滤人群` 弹窗和 `批量计划设置` 弹窗里。
- 原因：上一版为避免伪造未知接口，只打开原生入口或详情页，但这把“可批量操作”退化为逐个手动处理，且忽略了官方弹窗背后的稳定接口。
- 规则：当用户明确指出官方隐藏弹窗位置时，必须先抓该弹窗的读取/提交接口并接入官方链路；若未抓到接口，不得用跳转详情页伪装批量能力，应明确保守阻断并记录缺口。

## L51 Hover 弹层不能继承禁用态或陈旧菜单状态
- 触发：用户反馈 `批量+` 鼠标移上去仍看不到弹出菜单。
- 原因：复刻原生按钮后仍把原生禁用态当成交互禁用依据，并且菜单 DOM 被页面/测试移除时内部 `batchPlusMenuEl` 可能残留，导致后续 hover 被误判为“已打开”而直接返回。
- 规则：新增的扩展菜单若目标是补充原生禁用态下不可用的能力，触发器自身不能继承原生 disabled 交互；hover/click 入口必须按真实事件目标用 `closest()` 定位外层，并且所有菜单开关判断只信任仍连接在 DOM 上的菜单节点。

## L50 复刻原生弹层要复刻触发方式和尺寸约束
- 触发：用户继续指出 `批量+` 与 `批量计划设置` 的弹出逻辑不一致，并明确要求鼠标移动上去就弹出菜单，同时按钮宽度要适配 `批量+` 文案。
- 原因：上一版虽然克隆了按钮 DOM 和样式，但仍用点击切换菜单，并硬拷贝了原生 `批量计划设置` 的固定宽度，导致交互触发与视觉尺寸不符合原生使用感。
- 规则：复刻原生 popmenu 时必须同时核对触发事件、移入/移出关闭、菜单内悬停、按钮组间距和短文案宽度；克隆长文案按钮时不能默认继承固定宽度。外层占位也要跟随原生锚点的 computed layout（如 `float:left` 或 `inline-block`），不能跨页面一刀切。

## L49 复刻原生控件必须同构 DOM 与交互
- 触发：用户要求 `批量+` 的操作逻辑与 `批量计划设置` 一模一样，包括样式和弹出逻辑。
- 原因：第一版用自定义按钮和自定义下拉菜单近似实现，功能可用但没有复用原生控件的禁用状态、弹层结构和视觉细节。
- 规则：用户说“复刻/一模一样/参考原生”时，先抓真实页面 DOM、computed style、disabled 状态和弹层行为；优先克隆或同构原生结构，只替换必要文案与菜单项，不要用独立设计系统近似模拟。

## L48 有官方复制能力的业务线优先接原生复制接口
- 触发：用户反馈人群推广 `E7Pro_AI点睛_测试` 复制不完整，并指出人群推广原生页面已有官方复制计划功能。
- 原因：插件通用 `/solution/addList.json` 复制需要自行组装 campaign/adgroup 字段，难以完整覆盖原生弹窗级的人群、创意、灵犀调优等隐藏配置；官方 `/solution/copy.json` 只提交源计划 ID、新计划名和投放日期，由服务端复制完整设置。
- 规则：某业务线存在官方“复制计划”入口时，先抓取并接入官方复制接口，再叠加插件的命名、数量、暂停兜底和成功弹窗；不要优先用通用建计划 payload 复刻官方复制。

## L47 复制老计划不能保留与预算形态冲突的套餐支付字段
- 触发：用户反馈插件复制出的线索计划显示“余额支付”，而源计划和其它原生新建日预算计划不显示。
- 原因：复制旧线索日预算计划时保留了 `dmcType=normal + dayBudget`，但遗漏删除前置默认补齐的 `orderChargeType=balance_charge`；该字段属于套餐/订单型预算链路，日预算形态不应提交。
- 规则：复制计划时不仅要清理旧 ID 和时间字段，也要按目标预算/投放形态清理互斥字段；验证创建类字段异常时必须对比“源计划 + 原生新建计划 + 插件复制计划”的只读服务端详情，定位后端字段而不是只看页面文案。

## L46 创建类按钮验收必须确认服务端结果与目标状态
- 触发：用户反馈点击 `复制暂停` 后没有生成新计划；修复后又发现创建接口会忽略请求里的 `onlineStatus=0`，新计划初始仍为在投。
- 原因：早期只验证按钮点击、参数解析和 no-op 状态，未把真实 `/solution/addList.json` 返回、新 campaignId、后置状态查询作为完成标准。
- 规则：涉及“创建/复制/投放”类按钮时，授权后必须抓到创建接口成功响应并用只读查询确认新对象存在；若按钮语义包含暂停/开启等目标状态，还必须确认服务端最终状态与按钮文案一致，不能只看创建请求 payload。

## L45 行级操作按钮必须跟随原生操作区
- 触发：用户截图指出复制计划按钮应放在 `详情 / 报表 / 高级设置 / 置顶 / 更多` 后面，而不是计划名称或计划 ID 旁边。
- 原因：第一版复用了计划 ID 快捷入口的注入位置，虽然能拿到上下文，但视觉上把“复制计划”混入名称区，和用户认知中的行操作区不一致。
- 规则：新增行级业务动作时，优先定位原生操作按钮组并追加到操作组末尾；计划 ID 附近只放查数、并发开启这类与 ID 强绑定的小图标，不放文字型业务操作按钮。

## L44 小尺寸行内图标不能叠加多语义细节
- 触发：用户反馈计划 ID 快捷查数与并发开启两个行内图标“有点难看，而且不简约”。
- 原因：为表达“数据查询/并发启动”叠加文档线、列表线和启动符号，小尺寸下细节过密，违背功能图标“简练、清晰、缩小后仍可辨识”的要求。
- 规则：20px 左右行内热区的图标优先保留单一核心轮廓；若需要表达组合语义，只保留一主一次两个元素，禁止叠加文档、表格、层级、播放等三种以上视觉符号。

## L43 点击路径不能承担大包首次解析
- 触发：用户反馈“点击组建计划，就是卡死浏览器”。
- 原因：把 `keyword-plan-api` 主体从 extension 首包移到点击时懒加载后，首次点击需要同步加载、解析、执行约 2.8MB 关键词运行时，再立即渲染向导，主线程压力集中到用户点击路径。
- 规则：性能拆包不能把重解析成本转移到高频点击路径；未完成真实页面点击验收前，不允许交付“点击后首次加载大包”的方案。优先保证交互稳定，再考虑 idle 预热、真正模块化或更细粒度拆分。

## L1 详情列不能复用状态语义
- 触发：用户反馈“方案详情是获取到的方案介绍，怎么删掉了”。
- 原因：详情兜底使用了 `已开启/未开启`，把介绍文案覆盖。
- 规则：`状态` 只在状态列表达；`详情` 必须优先展示方案介绍或配置说明。

## L2 护航方案要保证“全量展示 + 状态独立 + 详情可读”
- 触发：连续多轮反馈“方案执行与手动设置不匹配”后，用户继续指出详情缺失。
- 原因：修状态列时引入了详情回退退化。
- 规则：涉及渲染结构调整时，必须同时做三项回归检查：
  - 全量方案是否都展示；
  - 状态是否仅影响状态列；
  - 详情是否保留介绍文案。

## L3 浏览器复测要覆盖真实提交链路
- 触发：仅看静态代码难发现“真实返回字段缺失时详情退化”。
- 原因：新链路返回内容不稳定，可能是完整设置也可能是默认兜底。
- 规则：每次修复后必须在真实 `one.alimama` 页面点击“立即扫描并优化”，以日志表格结果作为最终验收依据。

## L4 手动主开关未勾选也要保留“手动语义”
- 触发：用户反馈“手动设置里没开，但方案里预算/净投产比仍然执行”。

- 规则：当手动面板存在时，提交层必须读取 `includeDisabled` 快照；主开关未勾选时用“手动未勾选”设置提交，不得回退为默认执行方案。

## L5 浏览器复测前先确认运行态已刷新
- 触发：首次复测仍看到旧行为，刷新后才命中新分支。

- 规则：涉及 userscript 逻辑修复时，浏览器复测前必须先刷新页面并重新触发流程，再判定修复结果。

## L6 用户要求“新增”时默认并存，不能替换原方案
- 触发：用户明确纠正“不是删掉出价调控，是增加多一个净投产比调控”。

## L7 手动面板存在“总开关 + 子开关”时，提交语义必须与用户最近操作一致
- 触发：用户反馈“已勾选净投产比调控但执行不生效”。

## L8 提交流量摸排默认按“计划 × 分支”全覆盖执行
- 触发：用户补充“里面还有很多选择，每一个计划，每一选择提交一次并记录”。

## L9 提交流量记录要集中沉淀到 `addList.md`

- 规则：本轮及后续新建计划提交流量摸排，所有 URL、Header、Body、分支差异、阻塞点统一写入仓库根目录 `addList.md`；`tasks/todo.md` 仅保留计划、执行状态和复盘摘要。

## L11 用户指定“不要点某开关”时必须锁定操作边界

- 规则：当用户明确排除某个控件（如 `AI点睛`）时，后续操作必须把该控件视为禁区，仅在当前状态下继续覆盖其余分支，并在记录中标注“未触碰该开关”。

## L12 用户明确切换优先级时立即停止旧分支

- 原因：继续推进旧分支会偏离当前最高优先级，造成上下文污染和无效验证。
- 规则：当用户明确指定新的优先级或任务编号时，立即停止旧分支，把计划、实现和验证全部切到新任务；旧分支只保留必要背景，不继续执行。

## L13 目标专属原生控件不能只靠通用字段兜底

- 规则：涉及原生页面专属控件时，必须先在真实页面观察入口、弹窗、选择结果和提交字段，再按插件现有风格补 UI 与组包映射；不能仅凭已有 allowlist 或兜底字段判断已覆盖。

## L14 原生同构不仅看字段，还要对齐列表列结构
- 触发：用户继续指出“原生网页里趋势主题有3列”。

## L15 红榜能力要补“入口 + 切换逻辑”而不只补单个标签
- 触发：用户指出“只是添加了趋势红榜，没有效果红榜与流量红榜”。
- 原因：弹窗只覆盖了单一榜单语义，没有实现完整榜单组及切换行为。
- 规则：遇到原生多榜单场景时，必须同时补齐所有榜单入口，并实现切换后的候选排序/过滤逻辑；不能只加单个静态文案。

## L16 用户要求“并排可比”时，优先同屏多列而非单列切换
- 触发：用户明确要求“三列同时显示，方便选择，不采取切换tab的方式”。
- 原因：虽已实现三榜单切换，但交互路径仍需来回点击，和用户“同屏对比”的选择效率目标不一致。
- 规则：当用户明确要求同屏对比多个榜单时，候选区必须改为并排多列展示；保留统一搜索与添加逻辑，不再依赖 tab 切换单列渲染。

## L17 指标可视化诉求要与排序能力同步交付
- 触发：用户进一步要求“增加趋势指数”。
- 原因：仅有榜单排序但缺少关键指标可视化，用户无法快速理解每个主题的趋势强弱。
- 规则：当榜单依赖某核心数值排序时，列表需同步展示该指标列（本场景为`趋势指数`），确保“可选 + 可比 + 可解释”同时成立。

## L18 参考原生设计时要对齐“分榜指标语义 + 卡片信息层级”
- 触发：用户要求“参考原生的设计”，并给出三列红榜截图。
- 原因：仅实现三列并排和可选功能仍不足以满足“像原生”的预期，指标标题和视觉层级仍有差异。
- 规则：遇到“参考原生设计”类需求时，除功能同构外，还需对齐每列指标语义（如`趋势/投产/搜索`）和信息层级（标题、副文案、列表容器结构），再做验收。

## L19 多列榜单可读性优先级：先保主题列，再保辅助列
- 触发：用户反馈“现在看不到趋势主题的文字，缩小已选趋势的宽”。
- 原因：三列同屏后辅助列（指标/商品数/操作）和右侧已选面板占比偏大，挤压了核心主题文本区域。
- 规则：多列榜单布局中，主题名称必须是最高可读优先级；当空间冲突时先缩右侧面板和辅助列宽，并验证主题列最小可见宽度达标后再调视觉细节。

## L20 同一弹窗内输入框视觉要一致，避免局部蓝边破坏整体
- 触发：用户指出趋势主题弹窗里搜索框与手动添加框边框应与其它输入一致灰色。
- 原因：局部输入框继承了默认聚焦样式，和弹窗整体的灰色中性风格不一致。
- 规则：同一弹窗内同类输入控件应统一边框与聚焦视觉；若用户指定“和其它一样”，优先按页面主输入色阶统一处理。

## L21 同行控件对齐要覆盖“高度、行高、内边距”三个维度
- 触发：用户继续要求两个输入框高度与同一行按钮一致。
- 原因：仅改边框颜色无法保证视觉对齐，输入框和按钮在控件高度/行高上仍有差异。
- 规则：当用户要求“与某控件一样”时，需同步校准高度、行高和垂直内边距，避免只改单一属性导致仍不齐。

## L22 选择型列表改“勾选模式”时要同步重构已选展示容器
- 触发：用户要求“修改成打勾的方式，操作那一列可以删了”，并指定已选主题参考原生置顶浮窗样式。
- 原因：若只替换按钮为勾选而不改已选容器，交互路径仍割裂（勾选在列表、已选在侧栏），无法达到原生的一体化选择体验。
- 规则：当列表改为勾选模式时，需同步改造为“勾选即入选 + 置顶已选条 + 一键移除标签”，并移除冗余操作列与侧栏表格。

## L23 用户明确“浮在底部且不跟随滚动”时，不能用 sticky top 方案
- 触发：用户反馈“已选明星趋势主题 0/6 是浮动置顶在最下面，不跟着页面变动”。
- 原因：顶部 sticky 方案与原生底部悬浮行为不一致，滚动时位置感错误。
- 规则：此类需求应使用底部固定容器（absolute/fixed + 预留内容区），确保列表滚动时浮条稳定停留在底部。

## L24 弹窗内若要求“浮条始终可见”，必须避免整窗滚动
- 触发：用户反馈“已选明星趋势主题 0/6 没有悬浮固定，还是需要达到底部才显示出来”。
- 原因：弹窗容器整体滚动且只有 `max-height`，没有明确高度，导致内部 `height:100%` 不生效；底部浮条仍落在内容自然高度底部。
- 规则：此类弹窗需要设置明确窗口高度，并让 `body -> layout -> main` 逐层 `flex/height` 撑满；只改 `overflow:hidden` 不够。

## L25 原生三红榜是同一推荐接口的三个分榜字段
- 触发：用户反馈“效果红榜里投产指数没有数据”。
- 原因：原生 `recommendTheme.json` 同时返回 `trendThemeInfo / effectThemeInfo / capacityThemeInfo`；插件只读取第一个数组后用统一候选池排序模拟三榜，导致效果/流量红榜主题与原生不一致。
- 规则：三红榜必须按原生分榜字段分别渲染并保持原始顺序；`themeAssociation` 只能作为联想/兜底补充，不能替代原生红榜数据源。

## L26 原生红榜的右侧信息不能用等级字段替代
- 触发：用户指出红榜最右侧仍有缺失，并要求补手动搜索版块。
- 原因：趋势主题归一化时虽保留了 `trendLv`，但趋势红榜右侧原生语义是周增幅/趋势状态；若直接取 `trendLv.name` 会显示 `A` 这类等级，和原生视觉不一致。
- 规则：红榜右侧列必须按分榜语义取值：趋势榜优先周增幅/趋势状态，效果榜和流量榜取 `resourceType.tagText` 标签；新增原生版块时必须同步接入真实接口数据、选择状态与自动化测试，而不是只补静态 UI。

## L27 “悬浮固定”不是“贴底固定”
- 触发：用户纠正“是悬浮固定，不是底部固定”。
- 原因：此前把已选趋势条理解为贴近内容/弹窗底部固定，实际诉求是像原生一样作为悬浮层覆盖在内容上方，并且不随内容滚动。
- 规则：遇到“悬浮固定”时，应让组件脱离内容流，固定在弹窗可视层内的明确位置（通常在操作按钮上方），同时给可滚动内容留出避让空间；不要把组件作为普通内容放到底部。

## L28 原生联想入口不是静态筛选标签
- 触发：用户指出“关键词是可以输入的，本店宝贝弹出本店商品，竞店宝贝弹出搜索全网商品”。
- 原因：此前只把三类入口渲染成切换按钮和一个通用输入框，没有按原生分别实现输入、店内商品选择与全网商品搜索选择。
- 规则：复刻原生联想类交互时，必须区分入口的实际控件形态和数据来源；`关键词` 要能直接输入，`本店宝贝/竞店宝贝` 要能打开商品列表并用选中商品触发后续联想，不能用单个通用输入框替代。

## L29 竞店宝贝不能复用本店商品搜索接口
- 触发：继续验证后发现 `竞店宝贝` 复用 `material/item/findPage.json` 会返回本店/可投商品，和原生“搜索全网商品”不一致。

- 规则：看到“本店/竞店/全网”这类同形不同源入口时，必须分别抓接口和请求字段；只有本店商品可以走店内物料接口，竞店或全网商品应使用原生全网搜索接口并补回归断言。

## L30 宝贝标题不能直接等同趋势联想关键词
- 触发：用户反馈“选择本店宝贝，或者竞店宝贝，搜索后，下面趋势主题有结果，现在只有搜索关键词有结果”。

- 规则：宝贝驱动趋势联想时必须准备降级查询链路：先试完整标题，再提取商品核心词/类目词继续联想；快捷联想按钮也必须维护当前选中态，不能固定首个圆点。

## L31 新原生联想区替代旧入口时必须删除旧入口
- 触发：用户指出趋势主题弹窗里的 `.am-wxt-scene-crowd-native-manual` 多余，并要求联想区 `section > section` 高度 100%、不需要滚动。
- 原因：新增原生式关键词/本店宝贝/竞店宝贝联想后，旧“手动添加主题”入口已变成重复入口；同时联想区若按固定高度收缩，会让下方内容需要额外滚动才能看到。
- 规则：新交互替代旧入口时，必须同步删除旧 DOM、旧事件和回归测试覆盖；弹窗内的替代区应作为固定可视块撑满剩余空间，避免依赖整块滚动完成操作。

## L32 验证弹窗布局必须看可见内容，不只看 CSSOM
- 触发：用户指出联想区 CSSOM 是对的，但实际只看到一行，仍然看不全。
- 原因：只验证外层 `height/overflow` 会漏掉内部表格被红榜和悬浮条挤压的问题；视觉验收应看实际可见行数与滚动条，而不是只看计算样式。
- 规则：后续弹窗高度类修复必须用真实页面测量区域矩形、可见行数和滚动状态；若用户要求“不滑动看全”，优先重新分配上下区域高度，再判断 CSS 属性是否正确。

## L33 “页面可滚动”时不要继续压缩内部区域
- 触发：用户澄清“整个页面可以滚动，高度不限制，只是联想相关趋势主题下面的趋势主题需要显示全”。
- 原因：此前按固定弹窗高度强行压缩红榜和联想区，仍会造成联想结果区域只露少量行；用户实际接受整窗滚动，目标是取消联想区内部截断。
- 规则：当用户允许整页/整窗滚动时，应让内容自然展开，只保留外层滚动容器；不要在子区域设置固定高度或内部滚动条。

## L34 底部已选条在整窗滚动模式下必须用视窗级固定
- 触发：用户要求“每个红榜高度高5倍，已选明星趋势主题的模块是悬浮固定置顶在最下面”。
- 原因：弹窗允许整窗滚动后，内容内的 `absolute` 底部定位会随内容位置变化，不能保证始终停留在当前可视底部。
- 规则：整窗滚动 + 底部悬浮条的组合应使用 `fixed` 视窗级定位，并给内容区留出足够底部避让空间；列表高度类需求应按用户给出的倍率直接换算，不再额外压缩。

## L35 用户点名删除某条 CSS 时必须彻底移除固定值

## L52 AI 点睛详情开关必须用全局委托监听

## L53 AI 点睛常驻内容不能挂到详情折叠组

## L54 原生“查看和添加”入口要连接已有详情面板

## L56 Codex Apps wrapper 失效时先验证底层 MCP 端点

## L57 后台授权与本地短租约不能混同
- 触发：用户指出“没有授权过期，为什么老是会弹出提示呢”。
- 原因：授权服务后台授权仍有效，但 extension 本地租约 TTL 只有数分钟；若启动不从有效缓存恢复、到期前不静默续租，第一次点击会先被本地过期态阻断并弹提示。
- 规则：授权类提示必须区分“后台授权过期”和“本地短租约待刷新”；有效缓存应先恢复内存授权态并静默续租，只有远端明确未授权/真实过期时才展示锁定遮罩。

## L58 首装冷启动等待态不能渲染授权锁定页
- 触发：用户指出“新店铺刚安装时，也弹窗授权超时页面”。
- 原因：无本地租约缓存时，extension 冷启动和按需点击校验处在等待远端返回阶段；旧逻辑把 `license_checking` 和授权服务瞬时失败直接渲染成全屏锁定遮罩，导致用户误以为新店铺授权超时或已过期。
- 规则：授权等待态、店铺识别暂未完成和授权服务瞬时失败只能更新状态并阻断当前敏感事件，不渲染全屏锁定页；只有远端明确未授权、真实过期或非瞬时失败才调用 `lock()`。

## L59 跨浏览器运行态不能假设 localStorage 和 childList 永远可靠
- 触发：用户反馈“辅助显示没有生效，有些浏览器可以有些不可以”。
- 原因：主助手配置直接读写 `localStorage`，在隐私模式、存储受限或异常配置浏览器中可能抛错并阻断启动；同时 SPA 表格可能只修改 `class/style/aria-hidden` 或文本复用节点，单纯监听 `childList` 会漏掉重扫。
- 规则：主助手启动路径上的存储读写必须有异常兜底，失败时降级但不能阻断 UI 和辅助显示；DOM 增强类功能除新增/删除节点外，还要覆盖可见性属性和文本变化，并用真实页面刷新验证标签数量与开关交互。

## L60 “更新 N 项”不等于用户可见

- 规则：授权状态机必须区分“远端明确未授权/真实过期”和“扩展续租瞬时失败”；同步入口应优先恢复有效缓存或触发按需复验，只有非瞬时失败才锁定页面并清理缓存。

# 教训 - 2026-05-24 编辑计划入口修复必须同步产物

- 规则：用户说“窗口外/弹窗外/外面的背景”时，先定位对应 overlay/mask 的最终 computed style；只改该外层背景，除非用户明确要求，不改变弹窗本体背景和内部内容区。
