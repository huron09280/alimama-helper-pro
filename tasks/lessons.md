# Lessons - 2026-04-15

## L44 小尺寸行内图标不能叠加多语义细节
- 触发：用户反馈计划 ID 快捷查数与并发开启两个行内图标“有点难看，而且不简约”。
- 原因：为表达“数据查询/并发启动”叠加文档线、列表线和启动符号，小尺寸下细节过密，违背功能图标“简练、清晰、缩小后仍可辨识”的要求。
- 规则：20px 左右行内热区的图标优先保留单一核心轮廓；若需要表达组合语义，只保留一主一次两个元素，禁止叠加文档、表格、层级、播放等三种以上视觉符号。

## L43 点击路径不能承担大包首次解析
- 触发：用户反馈“点击组建计划，就是卡死浏览器”。
- 原因：把 `keyword-plan-api` 主体从 extension 首包移到点击时懒加载后，首次点击需要同步加载、解析、执行约 2.8MB 关键词运行时，再立即渲染向导，主线程压力集中到用户点击路径。
- 规则：性能拆包不能把重解析成本转移到高频点击路径；未完成真实页面点击验收前，不允许交付“点击后首次加载大包”的方案。优先保证交互稳定，再考虑 idle 预热、真正模块化或更细粒度拆分。

## L1 详情列不能复用状态语义
- 触发：用户反馈“方案详情是获取到的方案介绍，怎么删掉了”。
- 原因：详情兜底使用了 `已开启/未开启`，把介绍文案覆盖。
- 规则：`状态` 只在状态列表达；`详情` 必须优先展示方案介绍或配置说明。

## L2 护航方案要保证“全量展示 + 状态独立 + 详情可读”
- 触发：连续多轮反馈“方案执行与手动设置不匹配”后，用户继续指出详情缺失。
- 原因：修状态列时引入了详情回退退化。
- 规则：涉及渲染结构调整时，必须同时做三项回归检查：
  - 全量方案是否都展示；
  - 状态是否仅影响状态列；
  - 详情是否保留介绍文案。

## L3 浏览器复测要覆盖真实提交链路
- 触发：仅看静态代码难发现“真实返回字段缺失时详情退化”。
- 原因：新链路返回内容不稳定，可能是完整设置也可能是默认兜底。
- 规则：每次修复后必须在真实 `one.alimama` 页面点击“立即扫描并优化”，以日志表格结果作为最终验收依据。

## L4 手动主开关未勾选也要保留“手动语义”
- 触发：用户反馈“手动设置里没开，但方案里预算/净投产比仍然执行”。
- 原因：提交层在未勾选场景下回退到了诊断/默认设置，导致执行状态与手动面板不一致。
- 规则：当手动面板存在时，提交层必须读取 `includeDisabled` 快照；主开关未勾选时用“手动未勾选”设置提交，不得回退为默认执行方案。

## L5 浏览器复测前先确认运行态已刷新
- 触发：首次复测仍看到旧行为，刷新后才命中新分支。
- 原因：页面可能仍在旧脚本运行态，导致代码已改但行为未生效。
- 规则：涉及 userscript 逻辑修复时，浏览器复测前必须先刷新页面并重新触发流程，再判定修复结果。

## L6 用户要求“新增”时默认并存，不能替换原方案
- 触发：用户明确纠正“不是删掉出价调控，是增加多一个净投产比调控”。
- 原因：之前把 `bidConstraintValue` 直接改名为净投产比，语义被替换而非扩展。
- 规则：涉及方案扩展时必须先确认“并存/替换”语义；若用户说“增加”，则保留原键与原文案，再新增新键全链路映射与面板入口。

## L7 手动面板存在“总开关 + 子开关”时，提交语义必须与用户最近操作一致
- 触发：用户反馈“已勾选净投产比调控但执行不生效”。
- 原因：原逻辑要求总开关显式开启，子项勾选不会反向开启总开关，导致提交仍按“未勾选”。
- 规则：当用户勾选任一主子项时，总开关必须自动开启，确保视觉状态与提交语义一致；同时保留“全未勾选=不执行”的回归校验。

## L8 提交流量摸排默认按“计划 × 分支”全覆盖执行
- 触发：用户补充“里面还有很多选择，每一个计划，每一选择提交一次并记录”。
- 原因：若只按“每类计划抽一条”采样，会漏掉二级场景、卡位类型、商品选择模式等真正影响提交参数的分支。
- 规则：对新建计划类页面做提交流量摸排时，必须按“每个计划类型 × 每个可选分支”逐一模拟提交并留档；只有账号权限、商品条件或页面显式限制阻塞时，才允许记录为未覆盖，并注明前置条件。

## L9 提交流量记录要集中沉淀到 `addList.md`
- 触发：用户补充“所有的记录保存在 addList.md，后续开发可以直接查看”。
- 原因：若把参数散落在对话、`tasks/todo.md` 或临时笔记里，后续开发检索和对照成本高，也容易遗漏新增分支。
- 规则：本轮及后续新建计划提交流量摸排，所有 URL、Header、Body、分支差异、阻塞点统一写入仓库根目录 `addList.md`；`tasks/todo.md` 仅保留计划、执行状态和复盘摘要。

## L10 关键词推广要单独覆盖“添加关键词”入口
- 触发：用户指出“核心词设置里‘添加关键词’没有获取”。
- 原因：之前只抓了默认词包和卡位方式，没有把手动新增关键词这条独立入口当成单独分支处理。
- 规则：对 `onebpSearch` 这类依赖 `wordList` 的场景，除默认词列表外，必须额外覆盖“添加关键词 / 清空关键词 / 匹配方式切换”这类会直接改写 `wordList` 的入口；不能只记录默认推荐词结果。

## L11 用户指定“不要点某开关”时必须锁定操作边界
- 触发：用户纠正“自定义推广里面，不需要点击 AI点睛”。
- 原因：分支摸排时若按通用模板顺手切开关，容易触碰用户明确排除的入口，造成额外干扰和无效样本。
- 规则：当用户明确排除某个控件（如 `AI点睛`）时，后续操作必须把该控件视为禁区，仅在当前状态下继续覆盖其余分支，并在记录中标注“未触碰该开关”。

## L12 用户明确切换优先级时立即停止旧分支
- 触发：用户在继续未完成探索后明确要求“修复P1”。
- 原因：继续推进旧分支会偏离当前最高优先级，造成上下文污染和无效验证。
- 规则：当用户明确指定新的优先级或任务编号时，立即停止旧分支，把计划、实现和验证全部切到新任务；旧分支只保留必要背景，不继续执行。

## L13 目标专属原生控件不能只靠通用字段兜底
- 触发：用户指出“趋势明星里，没有原生网页中新增加趋势主题”。
- 原因：只修了通用目标字段与请求保留，没有把趋势明星专属的原生“新增趋势主题”交互纳入插件配置。
- 规则：涉及原生页面专属控件时，必须先在真实页面观察入口、弹窗、选择结果和提交字段，再按插件现有风格补 UI 与组包映射；不能仅凭已有 allowlist 或兜底字段判断已覆盖。

## L14 原生同构不仅看字段，还要对齐列表列结构
- 触发：用户继续指出“原生网页里趋势主题有3列”。
- 原因：已补齐趋势主题能力，但候选列表仍按 4 列展示，和原生视觉结构不一致。
- 规则：同构原生弹窗时，除字段和提交链路外，必须逐项核对表头/列数/操作位布局；发现不一致先修结构再做体验微调。

## L15 红榜能力要补“入口 + 切换逻辑”而不只补单个标签
- 触发：用户指出“只是添加了趋势红榜，没有效果红榜与流量红榜”。
- 原因：弹窗只覆盖了单一榜单语义，没有实现完整榜单组及切换行为。
- 规则：遇到原生多榜单场景时，必须同时补齐所有榜单入口，并实现切换后的候选排序/过滤逻辑；不能只加单个静态文案。

## L16 用户要求“并排可比”时，优先同屏多列而非单列切换
- 触发：用户明确要求“三列同时显示，方便选择，不采取切换tab的方式”。
- 原因：虽已实现三榜单切换，但交互路径仍需来回点击，和用户“同屏对比”的选择效率目标不一致。
- 规则：当用户明确要求同屏对比多个榜单时，候选区必须改为并排多列展示；保留统一搜索与添加逻辑，不再依赖 tab 切换单列渲染。

## L17 指标可视化诉求要与排序能力同步交付
- 触发：用户进一步要求“增加趋势指数”。
- 原因：仅有榜单排序但缺少关键指标可视化，用户无法快速理解每个主题的趋势强弱。
- 规则：当榜单依赖某核心数值排序时，列表需同步展示该指标列（本场景为`趋势指数`），确保“可选 + 可比 + 可解释”同时成立。

## L18 参考原生设计时要对齐“分榜指标语义 + 卡片信息层级”
- 触发：用户要求“参考原生的设计”，并给出三列红榜截图。
- 原因：仅实现三列并排和可选功能仍不足以满足“像原生”的预期，指标标题和视觉层级仍有差异。
- 规则：遇到“参考原生设计”类需求时，除功能同构外，还需对齐每列指标语义（如`趋势/投产/搜索`）和信息层级（标题、副文案、列表容器结构），再做验收。

## L19 多列榜单可读性优先级：先保主题列，再保辅助列
- 触发：用户反馈“现在看不到趋势主题的文字，缩小已选趋势的宽”。
- 原因：三列同屏后辅助列（指标/商品数/操作）和右侧已选面板占比偏大，挤压了核心主题文本区域。
- 规则：多列榜单布局中，主题名称必须是最高可读优先级；当空间冲突时先缩右侧面板和辅助列宽，并验证主题列最小可见宽度达标后再调视觉细节。

## L20 同一弹窗内输入框视觉要一致，避免局部蓝边破坏整体
- 触发：用户指出趋势主题弹窗里搜索框与手动添加框边框应与其它输入一致灰色。
- 原因：局部输入框继承了默认聚焦样式，和弹窗整体的灰色中性风格不一致。
- 规则：同一弹窗内同类输入控件应统一边框与聚焦视觉；若用户指定“和其它一样”，优先按页面主输入色阶统一处理。

## L21 同行控件对齐要覆盖“高度、行高、内边距”三个维度
- 触发：用户继续要求两个输入框高度与同一行按钮一致。
- 原因：仅改边框颜色无法保证视觉对齐，输入框和按钮在控件高度/行高上仍有差异。
- 规则：当用户要求“与某控件一样”时，需同步校准高度、行高和垂直内边距，避免只改单一属性导致仍不齐。

## L22 选择型列表改“勾选模式”时要同步重构已选展示容器
- 触发：用户要求“修改成打勾的方式，操作那一列可以删了”，并指定已选主题参考原生置顶浮窗样式。
- 原因：若只替换按钮为勾选而不改已选容器，交互路径仍割裂（勾选在列表、已选在侧栏），无法达到原生的一体化选择体验。
- 规则：当列表改为勾选模式时，需同步改造为“勾选即入选 + 置顶已选条 + 一键移除标签”，并移除冗余操作列与侧栏表格。

## L23 用户明确“浮在底部且不跟随滚动”时，不能用 sticky top 方案
- 触发：用户反馈“已选明星趋势主题 0/6 是浮动置顶在最下面，不跟着页面变动”。
- 原因：顶部 sticky 方案与原生底部悬浮行为不一致，滚动时位置感错误。
- 规则：此类需求应使用底部固定容器（absolute/fixed + 预留内容区），确保列表滚动时浮条稳定停留在底部。

## L24 弹窗内若要求“浮条始终可见”，必须避免整窗滚动
- 触发：用户反馈“已选明星趋势主题 0/6 没有悬浮固定，还是需要达到底部才显示出来”。
- 原因：弹窗容器整体滚动且只有 `max-height`，没有明确高度，导致内部 `height:100%` 不生效；底部浮条仍落在内容自然高度底部。
- 规则：此类弹窗需要设置明确窗口高度，并让 `body -> layout -> main` 逐层 `flex/height` 撑满；只改 `overflow:hidden` 不够。

## L25 原生三红榜是同一推荐接口的三个分榜字段
- 触发：用户反馈“效果红榜里投产指数没有数据”。
- 原因：原生 `recommendTheme.json` 同时返回 `trendThemeInfo / effectThemeInfo / capacityThemeInfo`；插件只读取第一个数组后用统一候选池排序模拟三榜，导致效果/流量红榜主题与原生不一致。
- 规则：三红榜必须按原生分榜字段分别渲染并保持原始顺序；`themeAssociation` 只能作为联想/兜底补充，不能替代原生红榜数据源。

## L26 原生红榜的右侧信息不能用等级字段替代
- 触发：用户指出红榜最右侧仍有缺失，并要求补手动搜索版块。
- 原因：趋势主题归一化时虽保留了 `trendLv`，但趋势红榜右侧原生语义是周增幅/趋势状态；若直接取 `trendLv.name` 会显示 `A` 这类等级，和原生视觉不一致。
- 规则：红榜右侧列必须按分榜语义取值：趋势榜优先周增幅/趋势状态，效果榜和流量榜取 `resourceType.tagText` 标签；新增原生版块时必须同步接入真实接口数据、选择状态与自动化测试，而不是只补静态 UI。

## L27 “悬浮固定”不是“贴底固定”
- 触发：用户纠正“是悬浮固定，不是底部固定”。
- 原因：此前把已选趋势条理解为贴近内容/弹窗底部固定，实际诉求是像原生一样作为悬浮层覆盖在内容上方，并且不随内容滚动。
- 规则：遇到“悬浮固定”时，应让组件脱离内容流，固定在弹窗可视层内的明确位置（通常在操作按钮上方），同时给可滚动内容留出避让空间；不要把组件作为普通内容放到底部。

## L28 原生联想入口不是静态筛选标签
- 触发：用户指出“关键词是可以输入的，本店宝贝弹出本店商品，竞店宝贝弹出搜索全网商品”。
- 原因：此前只把三类入口渲染成切换按钮和一个通用输入框，没有按原生分别实现输入、店内商品选择与全网商品搜索选择。
- 规则：复刻原生联想类交互时，必须区分入口的实际控件形态和数据来源；`关键词` 要能直接输入，`本店宝贝/竞店宝贝` 要能打开商品列表并用选中商品触发后续联想，不能用单个通用输入框替代。

## L29 竞店宝贝不能复用本店商品搜索接口
- 触发：继续验证后发现 `竞店宝贝` 复用 `material/item/findPage.json` 会返回本店/可投商品，和原生“搜索全网商品”不一致。
- 原因：本店宝贝与竞店宝贝虽然 UI 都是宝贝选择列表，但数据源不同；竞店侧应走 AI 全网商品搜索的 `similarItemQueryStr`，不能只取消本店 `tagId` 当作全网搜索。
- 规则：看到“本店/竞店/全网”这类同形不同源入口时，必须分别抓接口和请求字段；只有本店商品可以走店内物料接口，竞店或全网商品应使用原生全网搜索接口并补回归断言。

## L30 宝贝标题不能直接等同趋势联想关键词
- 触发：用户反馈“选择本店宝贝，或者竞店宝贝，搜索后，下面趋势主题有结果，现在只有搜索关键词有结果”。
- 原因：插件选中宝贝后只把完整宝贝标题传给趋势主题联想接口，长标题包含品牌、型号、规格和营销词，容易导致 `themeAssociation` 返回空。
- 规则：宝贝驱动趋势联想时必须准备降级查询链路：先试完整标题，再提取商品核心词/类目词继续联想；快捷联想按钮也必须维护当前选中态，不能固定首个圆点。

## L31 新原生联想区替代旧入口时必须删除旧入口
- 触发：用户指出趋势主题弹窗里的 `.am-wxt-scene-crowd-native-manual` 多余，并要求联想区 `section > section` 高度 100%、不需要滚动。
- 原因：新增原生式关键词/本店宝贝/竞店宝贝联想后，旧“手动添加主题”入口已变成重复入口；同时联想区若按固定高度收缩，会让下方内容需要额外滚动才能看到。
- 规则：新交互替代旧入口时，必须同步删除旧 DOM、旧事件和回归测试覆盖；弹窗内的替代区应作为固定可视块撑满剩余空间，避免依赖整块滚动完成操作。

## L32 验证弹窗布局必须看可见内容，不只看 CSSOM

## L51 AI 自动生成内容不能本地写死
- 触发：用户指出“插件中AI点睛怎么是固定写死的呢？原生网页里，关键词计划-自定义计划是添加商品后，如果开启AI点睛，会自动跑出来对应的设置”。

## L52 AI 点睛详情开关必须用全局委托监听

## L54 原生“查看和添加”入口要连接已有详情面板

## L56 Codex Apps wrapper 失效时先验证底层 MCP 端点

## L57 后台授权与本地短租约不能混同
- 触发：用户指出“没有授权过期，为什么老是会弹出提示呢”。
- 原因：授权服务后台授权仍有效，但 extension 本地租约 TTL 只有数分钟；若启动不从有效缓存恢复、到期前不静默续租，第一次点击会先被本地过期态阻断并弹提示。
- 规则：授权类提示必须区分“后台授权过期”和“本地短租约待刷新”；有效缓存应先恢复内存授权态并静默续租，只有远端明确未授权/真实过期时才展示锁定遮罩。

## L58 首装冷启动等待态不能渲染授权锁定页
- 触发：用户指出“新店铺刚安装时，也弹窗授权超时页面”。
- 原因：无本地租约缓存时，extension 冷启动和按需点击校验处在等待远端返回阶段；旧逻辑把 `license_checking` 和授权服务瞬时失败直接渲染成全屏锁定遮罩，导致用户误以为新店铺授权超时或已过期。
- 规则：授权等待态、店铺识别暂未完成和授权服务瞬时失败只能更新状态并阻断当前敏感事件，不渲染全屏锁定页；只有远端明确未授权、真实过期或非瞬时失败才调用 `lock()`。

## L59 跨浏览器运行态不能假设 localStorage 和 childList 永远可靠
- 触发：用户反馈“辅助显示没有生效，有些浏览器可以有些不可以”。
- 原因：主助手配置直接读写 `localStorage`，在隐私模式、存储受限或异常配置浏览器中可能抛错并阻断启动；同时 SPA 表格可能只修改 `class/style/aria-hidden` 或文本复用节点，单纯监听 `childList` 会漏掉重扫。
- 规则：主助手启动路径上的存储读写必须有异常兜底，失败时降级但不能阻断 UI 和辅助显示；DOM 增强类功能除新增/删除节点外，还要覆盖可见性属性和文本变化，并用真实页面刷新验证标签数量与开关交互。

## L60 “更新 N 项”不等于用户可见
- 触发：用户反馈日志里已经 `✅ 更新 60 项数据`，但网页中就是不显示。
- 原因：更新计数只代表 `.am-helper-tag` 创建或更新成功；若表格选择命中隐藏副本、旧 SPA 表格，或标签被单元格固定高度/overflow 裁剪，日志仍会显示成功但用户看不到。
- 规则：辅助显示、表格增强、DOM 注入类验收必须同时检查 DOM 写入数量、当前可见表格、可见标签数量和样式矩形；同页可能有上方汇总表和下方计划表，必须遍历全部可见指标表格，不能只更新排序最高的单个表格；必要时提供页面内诊断函数，不能只凭日志更新计数判定功能生效。

## L61 用户修正优先覆盖样式契约
- 触发：用户先说 `TAG_BASE_STYLE` 去掉强制 `width:100%` 的改动不要取消，随后纠正为“这个需要取消，我说错了”。
- 原因：样式契约直接影响用户熟悉的表格占位和视觉布局，不能把前一轮错误口径固化为测试和教训。
- 规则：当用户修正样式类决策时，必须同步更新源码、测试契约、任务文档和生成产物；历史教训只能保留仍成立的根因判断，不能保留与最新口径冲突的禁止项。

## L62 扩展续租瞬时失败不能降级为授权过期
- 触发：Chrome 扩展错误页出现 `[EscortAPI] 授权未通过，已阻止算法护航入口：授权已失效`，真实页面状态为 `lease_renew/request_failed`，但有效缓存和远端授权仍可恢复。
- 原因：extension 本地租约续租遇到 `Failed to fetch` 后进入瞬时失败态，算法护航同步门禁没有先从有效缓存恢复，也没有把 `request_failed` 视为可恢复等待态，最终误走 `lease_expired`。
- 规则：授权状态机必须区分“远端明确未授权/真实过期”和“扩展续租瞬时失败”；同步入口应优先恢复有效缓存或触发按需复验，只有非瞬时失败才锁定页面并清理缓存。

## L63 添加商品搜索必须先切到全部商品
- 触发：货品全站抓包时直接在默认推荐页按商品 ID 搜索无结果，用户纠正“添加商品，切换全部商品”。
- 原因：添加商品弹窗默认停留在推荐分组，商品 ID 搜索只作用于当前分组；未切到 `全部商品` 会误判商品不存在。
- 规则：货品全站或其它商品前置抓包中，按指定商品 ID 查找时必须先打开 `添加商品` 并切换 `全部商品`，再输入商品 ID 搜索；若商品可检索但不可推广，应记录页面阻塞原因，不改用“未找到”表述。

## L64 一级入口 baseline 不能替代内部控件覆盖
- 触发：完成 6 个一级场景 baseline 后，用户指出“需要这个才有用：每个内部按钮/开关/下拉全部覆盖”。
- 原因：入口级 `addList.json` 只能证明默认路径能组包，不能支撑开发还原页面内部开关、下拉、弹窗和高级设置的字段差异。
- 规则：新建计划抓包类任务的完成口径必须到控件级：每个可见按钮、开关、下拉、弹窗入口和高级设置都要有操作记录；能组包的分支记录 payload diff，不能覆盖的写明前置/阻塞/不点击原因。baseline 只能作为入口样本，不能标记全量任务完成。

## L65 开发向抓包必须额外保存完整请求体
- 触发：用户要求“这里提交的参数都记录下来，帮忙后续开发插件用”。
- 原因：仅记录摘要字段或关键键名只能辅助排查，不能直接支撑插件对齐真实接口结构、默认值、嵌套对象和可选字段；而且只保存“代表样本”会丢失不同场景和不同控件分支的真实组包差异。
- 规则：面向后续开发的抓包任务，除 `addList.md` 的摘要和 diff 外，还必须单独归档每一次实际触发到的 `addList` 请求 JSON，不能只保留代表样本。每条记录至少包含样本 ID、页面 URL、提交 URL、method、body length、完整 parsed body；未发请求的样本则必须记录阻塞原因，并与已成功样本分开保存。

## L66 设计体系改动必须沉淀成可执行规范
- 触发：完成图标重设计后，用户要求“保持成一个设计图标的规范”。
- 原因：只在源码和任务记录里体现设计决策，后续新增图标时仍可能回到 emoji、1024 iconfont、私有字体字符和硬编码颜色混用。
- 规则：涉及设计体系、交互模式或视觉语言的改动，除代码实现和测试外，必须沉淀到 `docs/` 下的规范文档，明确适用范围、实现入口、禁止项、验收清单和构建验证命令。

## L67 图标全量审计必须覆盖局部控件与 CSS 伪元素
- 触发：图标体系已建立后，二次审计仍发现关键词向导里的 `` 私有字体、`+`/`×`/`✕` 文本按钮、`?`/`P`/`*` 字符图标和 CSS 伪元素箭头/勾选。
- 原因：第一次验收重点覆盖主面板、扩展图标和共享 SVG 入口，没有把弹窗局部控件、矩阵编辑器、徽标计数、CSS `content` 与手绘勾选统一纳入扫描。
- 规则：后续“全量图标”任务必须同时跑源码残留扫描，覆盖私有字体字符、裸 `+`/`×`/`?`/`P`/`*`、`&times;`、`content:"⌄"` 这类 CSS 文本伪图标，以及 CSS 边框手绘勾；运行态可交互图标必须统一走共享 SVG 或明确记录为非图标文本内容。
# 教训 - 2026-05-24 编辑计划入口修复必须同步产物

- 触发：用户反馈“编辑计划无法编辑”，此前只改了 `src` 中入口短路，未同步确认生成产物与真实运行态。
- 原因：批量建计划向导真实运行可能来自根 userscript 或 extension bundle；只修改源码文件不能证明浏览器实际加载了新逻辑。
- 规则：修复 UI 入口类缺陷时，必须同时检查源码入口、生成产物同步和真实运行态；“编辑计划”这类打开入口不应兼具关闭 toggle 语义，关闭应由显式关闭按钮承担。
- 追加：用户复测仍不可编辑时，不能只验证“已进入编辑页”；必须在浏览器里真实修改字段并检查是否聚焦失败、输入回滚或保存不同步。

## L68 图标“统一”必须复用同一源路径而非相似重绘
- 触发：用户再次强调“悬浮的图标，谷歌插件的图标也用悬浮的图标”。
- 原因：之前的 Chrome 插件图标只改成相似闪电形状，未严格复用悬浮球 `renderAmIcon('logo')` 的源路径和圆形悬浮视觉。
- 规则：后续图标统一类任务必须先定位共享图标源，再用等比换算或共享渲染链路生成目标资产；回归测试要断言源路径关系，避免“看起来相似但不是同一图标”的实现。

### 2026-05-24 编辑计划弹层命中测试
- 对嵌套 modal/detail/backdrop 的 UI 修复，不能只看可见性或无障碍树；必须检查目标控件中心的 `document.elementFromPoint` 是否命中目标本身。
- `fixed` 子级放在带 `overflow:hidden` 的弹窗父级内时，即使视觉上显示，命中测试仍可能被父级裁剪并落到 sibling backdrop；优先用状态选择器精确放开 overflow，而不是粗暴禁用 backdrop pointer events。

### 2026-05-24 编辑计划重复表单
- 修复“不可编辑”时不能把隐藏的旧兼容表单直接显示出来作为解决方案；如果已有新的场景配置表单，应优先保证新表单可命中、可编辑，避免用户看到重复入口。

### 2026-05-25 弹窗浮层定位和图标背景
- 弹窗内小下拉、tooltip、菜单等浮层若要精确贴近触发器，优先采用 body 级 portal + `getBoundingClientRect()` 视口坐标定位，避免内部内容流、滚动容器和定位上下文继续造成错位。
- 图标按钮如果不需要按钮背景，必须显式覆盖 `background`、`border-color` 和 `box-shadow`，不能依赖通用按钮样式自然不泄漏。
- 小图标触发器不应挂原生 `title` 来表达状态；浏览器 tooltip 无法控制位置，容易被用户感知为错位。应使用 `aria-label` 保留语义，并在需要时自建受控 tooltip。
- 绝对定位在按钮内部的小图标若需要垂直居中，hover 态必须保留原始 `translateY(-50%)`，只叠加目标位移量；不能让通用按钮 hover 的 `transform` 覆盖居中 transform。

## L69 相邻分段控件需要显式行距
- 触发：用户指出矩阵页“场景选择 / 营销目标”上下可以相隔高一点。
- 原因：矩阵页场景卡片内两行分段控件高度较大，但行容器没有内部 gap，视觉上显得过于贴近。
- 规则：同一卡片内连续分段按钮行应在卡片容器层设置稳定 `gap`，不要靠按钮 padding 自然撑开；此类视觉修正应限定到目标卡片，避免影响通用表单行。

## L70 预览图确认后只落地确认范围
- 触发：用户在 UI 优化预览后确认“根据这个版本修改”。
- 原因：视觉稿确认容易被误扩成新增批量写入、提交链路或业务参数改造，增加真实投放风险和验收噪声。
- 规则：基于预览图实施时，先把“视觉重排 / 业务逻辑不变 / 禁止点击真实提交入口”写入任务计划；实现只改确认的结构与样式层级，批量编辑、提交、商品弹窗等既有业务入口必须复用原逻辑。

## L71 回退指令默认只回退最近一轮可识别改动
- 触发：用户在关键词向导首页二次贴图改版后要求“回退”。
- 原因：工作区同时存在多轮未提交改动，若直接按 git 文件回退会误伤上一版已验收首页 UI 和其它代理或用户改动。
- 规则：收到简短“回退”时，先限定为最近一轮可识别改动并按功能点做反向补丁；保留已验收前置版本和无关 dirty 文件，回退后仍需同步构建产物、测试和真实页面验证。

## L72 后台表格弹窗优先给足桌面宽度
- 触发：用户反馈关键词批量建计划首页“宽度可以更宽一点，就不会有文字的叠加”。
- 原因：计划表格同时承载计划名、多个标签、目标成本、预算和操作按钮，桌面弹窗宽度偏保守时即使 DOM 几何不重叠，视觉上仍会显得文字拥挤、标签堆叠。
- 规则：后台工作台类弹窗在桌面视口下应优先给表格足够宽度，再用窄屏媒体查询回退；不要只靠压缩字体、强截断或增加 title 解决可读性问题。

## L73 表格标签配色要绑定语义而非行号
- 触发：用户反馈关键词批量建计划首页“类型里，可以根据不同设置不同的颜色，同一个类型可以用深浅”。
- 原因：类型列多个标签共用单一蓝色，无法快速区分推广类型、营销目标和出价模式；若按行号或随机色处理，后续计划排序变化会造成认知不稳定。
- 规则：后台表格标签需要配色时，应按业务语义生成稳定 class，再用 CSS 变量表达色系和深浅；同一语义保持同色系，不同设置使用可区分但不过饱和的色彩。
