# Lessons

## 2026-04-15 用户修正：反馈“还是一样”时，先排除浏览器仍在运行旧扩展包
- 问题：代码与测试已修复，但浏览器若未重载扩展，页面仍会表现为旧逻辑，用户体感就是“还是一样生效了”。
- 纠偏规则：涉及扩展注入逻辑的复测，先执行 `chrome://extensions -> Reload`，再刷新业务页面后做验证，不得跳过版本重载步骤。
- 验证规则：复测结论必须附“重载后日志证据”，避免只依据未重载页面现象判定修复失败。

## 2026-04-15 用户修正：手动设置验收必须做主开关开/关双场景日志对照
- 问题：只测单一场景容易误判“默认全方案执行”或“手动设置未生效”。
- 纠偏规则：手动设置相关问题必须最少验证两条链路：`manual-enable=false` 与 `manual-enable=true`，并对照方案列表数量与名称。
- 验证规则：实页日志需同时出现：
  - 关闭主开关：`使用诊断返回设置`；
  - 开启主开关：`使用手动设置参数`（且方案数与勾选项一致）。

## 2026-04-15 用户修正：手动设置执行结果必须与启用项一一对应，禁止默认全方案误报
- 问题：手动设置与执行状态映射若未过滤 `enabled=false` 项，会出现“实际只开一个方案，但 UI/执行态看起来全方案都执行了”。
- 纠偏规则：openV3 执行状态生成只能基于真实启用项：优先 `operationList`，否则从 `userSetting` 严格筛选 `enabled !== false` 的键。
- 提交规则：主开关未勾选时，提交层必须退回默认护航参数；主开关勾选时，提交层必须按手动设置生成 `userSetting`，二者不可混用。
- 验证规则：实页必须完成双场景对照抓包：`manual-enable=false + 子项勾选` 与 `manual-enable=true + 单方案勾选`，并核对 `openV3.userSetting`。

## 2026-04-15 用户修正：手动设置辅助开关不得反向自动勾选“使用手动设置提交”
- 问题：把 `次日恢复` 等辅助 checkbox 纳入主从同步集合后，用户仅勾辅助项也会把主开关置为 `checked/indeterminate`，表现为“没勾主开关却自动生效”。
- 纠偏规则：手动设置面板必须拆分“主功能开关”与“辅助开关”；主开关状态仅由主功能开关反向驱动，辅助开关不能参与主开关启用态判断。
- 行为规则：主开关正向下发可继续同步主功能开关；辅助开关是否跟随主开关需显式定义，但绝不能影响主开关反向判定。
- 验证规则：实页至少验证两条链路：`仅勾辅助项 -> 主开关仍未勾选`、`勾主功能项 -> 主开关进入 checked/indeterminate`。

## 2026-04-14 用户修正：子账号展示应“一账号一行”，不要聚合成单条
- 问题：把同店铺下多个子账号合并成单条文本（如 `A、B、C`）会违背“一个子账号一条数据”的管理预期，影响逐条核对。
- 纠偏规则：涉及“记录全部子账号”需求时，默认按“明细行”实现（每个子账号独立一行），仅在用户明确要求汇总时才做聚合展示。
- 交互规则：即使多个子账号归属于同一 `shopId`，列表仍应拆分显示；授权/吊销操作可继续按 `shopId` 作用，但行展示必须是子账号粒度。
- 验证规则：至少验证一组同店多子账号样本，确认表格行数与子账号数一致，且每行 `subAccount` 字段为单值。

## 2026-04-14 用户修正：openV3 显式状态采集必须排除辅助控件，避免误开启
- 问题：显式状态采集若把所有 `checkbox/radio/aria-checked` 一并视为“启用态来源”，`次日恢复` 或 `匹配方式` 等辅助控件会把未开启方案误判为 `enabled=true`。
- 纠偏规则：启用态采集必须先过滤辅助控件（至少排除 `radio` 与 `dailyReset/keywordLimit/preference/lower/upper/times/targetName` 字段控件），仅保留主开关候选。
- 稳定性规则：同一 key 命中多个控件时，不允许后续控件覆盖首个主开关状态，避免辅助控件反向污染。
- 验证规则：新增/修改显式状态逻辑后，必须补充契约断言覆盖“辅助控件过滤 + radio 排除 + 同 key 不覆盖”。

## 2026-04-14 用户修正：主文案在折叠态下应承担“展开”语义，展开后才恢复“选择”语义
- 问题：将主开关文案放在 `label` 中时，折叠态点击文案会直接触发勾选，用户期望却是先展开内部设置再决定是否选择。
- 纠偏规则：对“折叠体 + 主 checkbox”组合，若要求“先展开再选择”，必须在折叠态拦截 `label` 默认行为（`preventDefault + stopPropagation`），仅执行展开。
- 行为规则：该拦截只在折叠态生效；展开态必须放行默认 `label/checkbox` 行为，避免破坏常规勾选体验。
- 验证规则：实页至少验证三态链路：`before(折叠未选)` -> `点文案后(展开未选)` -> `再点文案(展开已选)`。

## 2026-04-14 用户修正：折叠入口优先图标化，点击行为必须是“展开”而非“选择”
- 问题：手动设置头部若使用“展开设置”文字按钮，会显得重；且若展开入口与选择控件语义混淆，容易误触发勾选。
- 纠偏规则：在“主开关 + 折叠体”结构中，展开入口优先采用紧邻文案的小图标（如 `▸/▾`），视觉简约且语义清晰。
- 行为规则：展开入口点击只允许改变折叠状态，不得写入主开关值；主开关勾选状态只能由主开关自身或明确同步逻辑改变。
- 验证规则：至少覆盖两类断言：`图标态样式存在`、`不再出现展开文字按钮`；并在实页验证“点击图标前后 master checked 不变”。

## 2026-04-14 用户修正：小万护航手动设置默认应折叠，按需展开
- 问题：手动设置区域默认全量展开会增加面板噪音，干扰主流程；用户希望按需查看内部项。
- 纠偏规则：高频面板中的“高级/手动配置”默认折叠，仅保留主开关与明显的“展开设置”入口；点击后再显示详细字段。
- 交互规则：折叠状态必须可显式切换（`aria-expanded`/`hidden` 同步），并在“用户已开启主开关”或“从弹窗带入配置”时自动展开，降低误操作概率。
- 验证规则：新增源码契约测试，至少覆盖“默认折叠 DOM 状态”和“展开绑定函数/默认绑定调用”两类断言。

## 2026-04-14 用户修正：小万护航条件启用态必须只信任显式控件，禁止“根文本命中即开启”
- 问题：弹窗解析若先按整块文本命中 alias 直接置 `enabled=true`，在用户未勾选条件时也会被误判为生效并提交到 openV3。
- 纠偏规则：护航设置项启用态只能来自显式控件（`checkbox/radio/aria-checked`），且 key 匹配必须基于控件最近语义容器，禁止使用整弹窗根文本推断启用。
- 回退规则：当“本次弹窗存在显式状态”但某项未识别到显式值时，默认关闭；只有“全量无显式状态”才回退平台 `operationList` 兼容值。
- 验证规则：每次触达 `readSettingFromModal` 后，必须覆盖两条断言：`移除根文本默认开启逻辑` 与 `显式优先+默认关闭+兜底回退链路`。

## 2026-04-13 用户修正：shopId识别要“主键优先 + 降级校验”，授权维度要按店铺名称聚合
- 问题：当前识别链路把 `seller/member/merchant/owner` 等字段与 `shopId` 同权处理，易在多账号场景引入“同店铺名多 ID”噪声；同时管理台按单 `shopId` 授权会把同店不同子账号割裂。
- 纠偏规则：前端店铺标识识别必须“`shopId/shop_id` 优先命中”；其他 ID 字段只能降级候选，需结合同源 `shopName`/URL/请求上下文做二次校验后才可采用。
- 授权规则：涉及授权有效期时，默认按“店铺名称维度”聚合展示与授权（子账号仅作展示标识），避免同店子账号授权时间不一致。
- 验证规则：每次修改后必须完成两类验证：真实阿里妈妈页面 `chrome-devtools` 识别证据 + 管理台授权到期在同店多子账号间一致性验证。

## 2026-04-13 用户修正：手动设置勾选默认全关且主从双向同步
- 问题：若默认勾选为开启，且外层“使用手动设置提交”不与内层勾选联动，会造成用户理解和实际提交状态不一致。
- 纠偏规则：手动设置区域 checkbox 默认值统一为关闭；外层 master checkbox 必须与内层选项双向同步（master 改变驱动 children，children 变化反写 master，部分选择显示半选态）。
- 验证规则：每次改动该区域后，至少覆盖两条契约测试：`默认全关` 与 `主从同步（含 indeterminate）`。

## 2026-04-13 用户修正：小万护航手动设置复选框必须显式兜底可见样式
- 问题：手动设置面板复选框若仅依赖浏览器默认样式，在部分页面全局 reset/重绘规则下会被隐藏，表现为“没有打勾选择”。
- 纠偏规则：凡是注入到阿里妈妈页面的配置面板，`input[type="checkbox"]` 必须提供作用域内可见性兜底（至少包含 `appearance:auto !important`、`display:inline-block !important`、`visibility:visible !important`）。
- 验证规则：新增/修改注入式 checkbox 后，至少补一条源码契约测试，断言可见性兜底与 disabled 态视觉样式同时存在。

## 2026-04-11 用户修正：高频操作日志禁止直出超长计划名
- 问题：复制操作日志若直接拼接 `removedOriginal.name`，在“复制”链式命名场景会形成超长文本，日志区可读性显著下降。
- 纠偏规则：高频日志（复制/批量操作）优先使用结构化短文案（数量 + 状态），不要直出可无限增长的名称字段。
- 验证规则：复制动作日志需满足两点：单条日志不包含原计划完整长名；同一批次关键信息仍可通过“复制数量 + 是否删原计划”判断。

## 2026-04-11 用户修正：复制计划阈值与命名递增必须一致
- 问题：将“复制后删原计划”阈值写成 `>2` 会与用户期望（`>=2`）不一致；同时若复制命名先复用原后缀再走唯一性兜底，容易生成 `..._1_2` 这类嵌套后缀。
- 纠偏规则：复制动作需求涉及“阈值”时，必须逐字对齐比较符（`>`/`>=`/`<`）；涉及“序号递增”时，优先使用“解析原序号 + 连续递增”的命名策略，禁止依赖冲突后补 `_2` 的嵌套后缀兜底。
- 验证规则：浏览器验证至少覆盖两条：`copyBatchCount=2` 与 `copyBatchCount=3` 都会删除原计划；复制后计划名不存在 `/_\\d+_\\d+$/` 嵌套后缀。

## 2026-04-11 用户修正：管理台出现 Failed to fetch 时先做浏览器/阿里云链路排查
- 问题：若直接在前端代码层面修补日志，而不先核验云端域名连通性（DNS/TLS/CORS），容易把环境链路故障误判为代码缺陷。
- 纠偏规则：当用户要求“先查原因”或日志出现 `Failed to fetch` 时，先完成三段式取证：`DNS 对比（本地解析 vs 公网权威） -> TLS 握手 -> API 探测`，确认链路层问题后再决定是否改代码。
- 验证规则：排查结论必须至少包含两类证据：`dig/nslookup` 结果与 `curl/openssl` 握手或请求结果，并在 `tasks/todo.md` 复盘中记录。

## 2026-04-11 用户修正：已明确取消的需求必须立即停止并移除进行中计划项
- 问题：在规划模式中若继续保留已被用户取消的 TODO，会造成后续误执行与上下文噪声，影响任务优先级判断。
- 纠偏规则：用户明确说“删掉/不需要做了”后，必须立即终止该分支并从 `tasks/todo.md` 移除对应进行中条目（或显式标记取消），不得继续推进实现或评审。
- 验证规则：收到取消指令后，至少校验两项：`tasks/todo.md` 中对应条目已删除/取消、当前会话输出已声明该分支终止。

## 2026-04-09 用户修正：货品全站批量建计划必须保持“插件计划名 == 实际创建计划名”
- 问题：全站场景提交链路若把非英数计划名强制改写为 `site时间戳`，会导致用户在插件里配置的计划名称与后台实际创建名称不一致，影响对账与复盘。
- 纠偏规则：计划名以用户输入/插件生成值为准，禁止在场景分支中静默改写为随机名；仅允许空值兜底与必要的空白清洗。
- 验证规则：每次改动全站 `buildSolutionFromPlan` 或创建提交流程后，必须校验两项：`campaignName` 透传保持计划名、回归测试中不存在 `siteNameAutoNormalized` 特判。

## 2026-04-08 用户修正：省份/城市阶段禁止重复触发同周期渐显，避免上方闪烁
- 问题：增量渲染中若同一周期在后续任务完成时重复附加渐显，会让上方区域出现连续闪烁，影响阅读稳定性。
- 纠偏规则：周期列渐显必须做去重控制（单次加载中每个周期最多触发一次）；后续同周期数据更新仅更新内容，不重复动画。
- 验证规则：真实页面验证需统计渐显 episode，确认 `3/7/30/90` 各周期最多 1 次且 `duplicatePeriods=[]`。

## 2026-04-08 用户修正：人群看板增量显示默认要有平滑过渡，避免“突兀跳变”
- 问题：增量渲染虽然提升了“边加载边可见”，但新行/新列直接硬切会产生明显跳变，影响可读性和进度感知。
- 纠偏规则：对增量渲染默认增加短时、定向的渐显动画；仅作用于“本次新增周期列”，禁止整表闪烁。
- 验证规则：每次改动增量渲染链路后，必须在真实页面确认三件事：`loading 状态推进`、`is-progressive-reveal + revealIndex` 注入、`animationName/Delay` 生效。

## 2026-04-07 用户修正：扩展安装默认路径应优先“解压加载”，不要引导 CRX 直装
- 问题：本地生成 `.crx` 后直接安装，在 Chrome 非商店侧载场景会被策略拦截，用户表现为“扩展开关无法正常开启”。
- 纠偏规则：安装指引默认统一为“`dist/extension/` 或 zip 解压目录 -> 加载已解压扩展程序”；`CRX` 仅作为工件，不作为默认安装路径。
- 落地规则：仓库需提供本地一键 zip 命令，并在 README 明确写出“不要直接安装 `.crx`”的限制说明。

## 2026-04-07 用户修正：按需授权不能让“最近活动”停更
- 问题：extension 授权改为按需后，若在租约有效时直接跳过远端 `verify`，管理台 `activeShops.lastSeenAt` 会停在历史时间，运营误判为“店铺不活跃”。
- 纠偏规则：按需模式下，交互触发应保留“有限频率的远端刷新”能力（如节流窗口 + 强制校验），不能只依赖本地租约缓存。
- 验证规则：每次调整授权触发策略后，必须校验“插件交互 -> `/v1/license/verify` -> `admin/state.activeShops.lastSeenAt` 可持续更新”链路，并补测试断言防回退。

## 2026-04-07 用户修正：无域名线上可视化应优先走第三方默认域名托管
- 问题：在“不能购买域名”的约束下，继续围绕 OSS/FC 默认 HTML 入口排障会反复撞上平台下载策略，投入产出低。
- 纠偏规则：直接切到“前端静态托管（GitHub Pages/Vercel/Netlify 默认域名）+ FC 默认域名 API”架构，避免再以 OSS 默认 HTML 直开为交付目标。
- 落地规则：仓库必须同时维护三平台配置（Pages 工作流、`vercel.json`、`netlify.toml`）和对应文档说明，确保可随时替换托管商。

## 2026-04-07 用户修正：默认域名可用性判断必须先对照 OSS 官方策略
- 问题：遇到“默认域名访问 HTML 变下载”时，若只从代码层反复排查，会误把平台安全策略当成服务缺陷，导致修复方向偏离。
- 纠偏规则：先核对阿里云官方文档《如何配置访问OSS文件时的预览行为》，并用响应头特征做判定：`Content-Disposition: attachment` + `x-oss-force-download: true`（常伴随 `x-oss-ec: 0048-00000001`）。
- 落地规则：在“未购买域名”约束下，不再承诺默认域名直接渲染；改用“本地管理页 + 默认 FC API”作为可交付通路，并在浏览器完成通过链路验收。

## 2026-04-03 用户修正：授权台 X 删除后必须按“新店铺重置”语义
- 问题：仅删除单层记录或仅做 UI 隐藏，会导致店铺重连时仍命中历史授权/吊销状态，和“删掉后重新当新店”预期不一致。
