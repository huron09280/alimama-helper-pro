本文档面向维护者，帮助在当前仓库中修改 `src/` 并生成 [`阿里妈妈多合一助手.js`](../阿里妈妈多合一助手.js) 时快速建立上下文。

适用范围：

- 当前仓库主线代码
- 主脚本头版本：`6.05`
- 维护重点：双 IIFE 架构、多文件源码生成链、网络 Hook、万能查数、人群矩阵、计划 ID 快捷入口、批量建计划 API、算法护航、MV3 插件兼容层

## 1. 开发入口

这个仓库现在有轻量构建层：`src/` 是源码事实来源，根目录的 `阿里妈妈多合一助手.js` 是由 `node scripts/build.mjs` 生成的生产脚本产物。绝大多数源码仍保持原有双 IIFE 结构，只是被切分到多个功能文件中维护；仓库新增了零依赖的 `package.json` 脚本入口，便于 Codex 与新维护者快速发现常用命令。

```bash
npm run build
npm run build:check
```

本地开发最常用的命令：

```bash
npm run build
npm run build:check
npm run check:syntax
npm run codex:map
npm run codex:find -- "__AM_HOOK_MANAGER__"
npm run codex:changed
npm run test
npm run review
npm run dev:serve
```

底层等价命令仍然可用：

```bash
node scripts/build.mjs
node scripts/build.mjs --check
node --check "阿里妈妈多合一助手.js"
node --test tests/*.test.mjs
bash scripts/review-team.sh
```

联调方式：

1. 修改 `src/` 后执行 `node scripts/build.mjs`
2. 在根目录启动 `npm run dev:serve`（或 `python3 -m http.server 5173`）
3. 把 [`dev/dev-loader.user.js`](../dev/dev-loader.user.js) 装到 Tampermonkey

1. 执行 `node scripts/build.mjs`

3. 在阿里妈妈真实页面验证 page bundle 注入与面板能力

## 2. 架构总览

运行时代码仍采用双 IIFE 结构，这个边界不要轻易打散；当前变化主要是源码文件从单文件迁到 `src/`，再由构建脚本顺序拼装回根文件。

### IIFE 1：页面增强主助手

职责包括：

- 主面板与辅助显示
- 表格指标增强与花费排序
- 下载拦截弹窗
- 万能查数弹窗与人群对比看板
- 计划 ID 自动识别与快捷查数/并发开启
- 潜力词计划日维度 CSV 导出
- 统一网络 Hook 管理器

### IIFE 2：算法护航 + 建计划 API

职责包括：

- Token 捕获与 API 请求封装
- 算法护航诊断与提交
- 无界批量建计划 API
- 关键词/人群/全站等场景向导
- 页面桥接 API 暴露

### 共享约束

- 两个 IIFE 都通过 `globalThis.__AM_GET_SCRIPT_VERSION__` 取版本，不能改回硬编码
- 两个 IIFE 共用统一 Hook 管理器 `window.__AM_HOOK_MANAGER__`
- 两个 IIFE 之间的通信靠全局桥接，不靠闭包共享

## 6. 关键执行链路

### 6.1 页面增强主循环

入口在 `main()`：

1. `UI.init()`
2. `BudgetFrontendLimitBypass.init()`
3. `Interceptor.init()`
4. `CampaignIdQuickEntry.init()`
5. `PotentialPlanDailyExporter.init()`
6. 启动 `MutationObserver`
7. 调 `scheduleRunCore()`，最终执行 `Core.run()`

`Core.run()` 负责：

- 识别当前有效表格
- 计算/注入指标标签
- 自动执行花费排序

注意：

- 当关键词向导、场景配置弹窗、万能查数弹窗、并发日志弹窗等覆盖层可见时，`shouldPauseInjectionForPopup()` 会阻止主注入
- 相关选择器是测试契约的一部分，改动前先看 `tests/ui-popup-injection-pause.test.mjs`

### 6.2 网络 Hook 流程

统一 Hook 由 `createHookManager()` 创建，只允许安装一次。

主要消费者：

- `Interceptor.registerHooks()`：下载链接捕获
- `TokenManager.hookXHR()`：从 XHR URL/Body 中补抓 token
- `recordApiRequestToHookHistory()`：手动把 fetch 请求写回 hook 历史

这里的原则是：

- 不要再各自 monkey patch `fetch`/`XMLHttpRequest`
- 新增网络侧能力优先注册到现有 Hook 管理器

### 6.3 万能查数与人群对比看板

主入口在 `MagicReport`：

- `openWithCampaignId(...)`
- `switchMagicView(...)`
- `queryCrowdInsight(...)`
- `requestCrowdApi('/ai/report/dataQuery.json' / 'panelDataQuery.json', ...)`

当前默认视图来自 `State.config.magicReportDefaultView`，默认值是 `matrix`。这部分改动通常会影响：

- 弹窗结构
- 默认 tab 逻辑
- 人群对比看板指标与周期显隐
- iframe 清理规则

对应回归：

- `tests/magic-report-crowd-matrix.test.mjs`

### 6.4 计划 ID 快捷入口

主入口在 `CampaignIdQuickEntry`：

- 页面扫描计划 ID 文本
- 注入快捷查数按钮
- 注入“并发开启”按钮
- 通过计划维度和商品维度做缓存与反查

并发开启相关逻辑也在这里，包括：

- 冲突识别
- 同商品计划联动
- 全站 + 自定义同开突破策略
- 日志弹窗

对应回归：

- `tests/campaign-concurrent-start-quick-entry.test.mjs`

### 6.5 潜力词计划 CSV 导出

主入口在 `PotentialPlanDailyExporter`：

- 只在 `#/report/bidword/index?...mainTab=potential` 生效
- 先取计划，再取单元，再按天拉表格
- 最终组装 CSV 并触发下载

对应回归：

- `tests/potential-plan-daily-exporter.test.mjs`

### 6.6 算法护航流程

护航执行链路在 IIFE 2 的 `Core.processCampaign(...)` 一带：

1. `TokenManager.refresh()`
2. 调 `https://ai.alimama.com/ai/chat/talk.json`
3. 从响应中递归收集 `actionList`
4. 找出“算法护航”方案
5. 调 `https://ai.alimama.com/ai/escort/open.json` 提交
6. 用护航面板卡片展示过程日志和结果

主面板通过 `window.__ALIMAMA_OPTIMIZER_TOGGLE__` 拉起这套 UI。

### 6.7 建计划 API 与向导

入口是 `KeywordPlanApi`，它对外暴露的方法包括：

- `openWizard`
- `getRuntimeDefaults`
- `searchItems`
- `createPlansBatch`
- `createPlansByScene`
- `createSitePlansBatch`
- `createKeywordPlansBatch`
- `createCrowdPlansBatch`
- `createShopDirectPlansBatch`
- `createContentPlansBatch`
- `createLeadPlansBatch`
- `runCreateRepairByItem`
- `appendKeywords`
- `suggestKeywords`
- `suggestCrowds`
- `validate`
- `getSessionDraft`
- `clearSessionDraft`

这里的维护重点：

- `wizardState` 同时承载 UI 与请求构建态
- 默认场景配置是 API-only 模式，`WIZARD_FORCE_API_ONLY_SCENE_CONFIG = true`
- 场景切换、营销目标、出价方式、预算类型、关键词模式、资源位/地域/时段设置都会影响最终请求体
- 不同场景有不同的 item 绑定规则、预算回填规则、提交模式

核心请求流通常是：

1. `openWizard()` 打开向导
2. `buildRequestFromWizard()` 组装请求预览
3. `createPlansByScene(sceneName, request, options)` 提交
4. 内部通过 `requestOne()` / `requestOneGet()` 调 `one.alimama.com` 接口

如果改动的是关键词/人群/全站场景逻辑，优先先读相关测试，再动代码。

## 9. 高频风险清单

### 9.1 不要重复 patch 网络层

已经有 `createHookManager()` 统一管理 `fetch` 和 `XMLHttpRequest`。如果在新功能里再次直接 patch，很容易造成：

- 重复上报
- token 捕获失效
- 下载拦截多次触发

### 9.2 不要硬编码版本号

版本显示必须走：

```js
globalThis.__AM_GET_SCRIPT_VERSION__()
```

不要手写 `v6.xx` 到 UI、日志或标题里。

### 9.3 不要随便改桥接全局名

跨 IIFE、跨页面上下文、跨测试都依赖这些名字。改动前先全局搜索 `__AM_` 和 `__ALIMAMA_`。

### 9.4 谨慎处理弹窗与注入并发

当前主注入在弹窗打开期间会暂停。新增覆盖层时，如果会遮挡主页面或影响表格结构，应把选择器补进 `shouldPauseInjectionForPopup()`。

### 9.5 注意配置源不止一套

- 主助手：`localStorage`
- 护航：`GM_getValue('config')`
- 向导草稿：`sessionStorage`

排查“刷新后丢配置”“面板显示和提交不一致”时，先确认查的是哪一套。

### 9.6 不要把双 IIFE 内部同名对象搞混

文件里 `Logger`、`State`、`UI`、`Core` 都在两个 IIFE 中各自定义了一套。搜代码时一定要先确认自己在第一个 IIFE 还是第二个 IIFE。

## 11. 建议的改动顺序

1. 先找模块锚点，确认改动属于哪个 IIFE
2. 再看对应测试，确认哪些契约不能破
3. 小步修改 `阿里妈妈多合一助手.js`
4. 至少跑 `node --check` 和相关功能测试
5. 涉及 UI / 提交 / 配置同步时，再按 [`other/SMOKE_TEST_CHECKLIST.md`](../other/SMOKE_TEST_CHECKLIST.md) 做手工回归

如果只是新增功能，不确定应该挂在哪个模块，优先遵循现有边界：

- 页面增强、查数、下载、快捷入口放 IIFE 1
- token、护航、建计划 API、页面桥接放 IIFE 2
