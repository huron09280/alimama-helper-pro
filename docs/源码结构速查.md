# 源码结构速查

这份文档的目标只有一个：下次要改功能时，先看这里，5 分钟内知道应该去哪一个文件夹、哪个文件。

## 先记住三条

1. `src/` 是源码事实来源，业务代码都应该优先改这里。
2. 根目录 `阿里妈妈多合一助手.js` 是生成产物，不再手工维护。
3. 改完 `src/` 后先跑 `npm run review`（或底层等价命令），再做浏览器验证。

## 入口优先级

1. 第一次进入仓库先看 `README.md` 的“5 分钟上手”。
2. 已经知道要改哪个功能时再看这份速查文档。
3. 完成改动后统一执行 `npm run review` 作为门禁入口。

## 常用命令

推荐入口（npm scripts）：

```bash
npm run build
npm run build:check
npm run check:syntax
npm run test
npm run review
```

底层等价命令：

```bash
node scripts/build.mjs
node scripts/build.mjs --check
node --check "阿里妈妈多合一助手.js"
node --test tests/*.test.mjs
bash scripts/review-team.sh
```

如果你已经大概知道要改什么，但还不确定具体函数在哪，先用下面这些关键词搜。

### 通用搜法

```bash
rg -n "关键词" src tests
rg -n "关键词" src/optimizer/keyword-plan-api tests
rg -n "关键词" "阿里妈妈多合一助手.js"
```

建议顺序：

1. 先搜 `src/`，找到真实源码入口。
2. 再搜 `tests/`，看这个功能有没有现成回归。
3. 只有在排查构建产物或兼容性问题时，才搜根文件 `阿里妈妈多合一助手.js`。

## `src/` 目录说明

### `src/entries/`

构建入口层，决定 userscript 和 extension 的装配方式。

- `src/entries/userscript-meta.js`
  - userscript 头部信息，含 `@name`、`@match`、`@grant`、`@version`。
- `src/entries/extension-content.js`
  - MV3 content script，只负责把页面 bundle 注入主世界。
- `src/entries/extension-page-compat.js`
  - extension 兼容层，补齐页面环境下的 `GM_*` 能力和版本桥接。

### `src/shared/`

共享前置层。

- `src/shared/script-preamble.js`
  - 全局前置脚本，负责版本读取、共享前导逻辑和双 IIFE 通用能力。

### `src/main-assistant/`

页面增强主助手，偏“投放后台增强”和“表格/面板/下载/查数”。

- `src/main-assistant/bootstrap.js`

- `src/main-assistant/ui.js`

- `src/main-assistant/budget-frontend-limit-bypass.js`
  - 预算前端限制绕过相关逻辑。
- `src/main-assistant/interceptor.js`

## 开发建议顺序

1. 先在这份文档找到要改的模块文件。
2. 再用 `rg` 搜方法名或 UI 文案，确认调用链。
3. 修改 `src/` 后运行 `node scripts/build.mjs`。
4. 跑对应测试，最后跑 `bash scripts/review-team.sh`。
5. 涉及 UI、Hook、建计划、下载链路时，再做真实阿里妈妈页面验证。

## 不建议手改的区域

- `阿里妈妈多合一助手.js`
  - 这是生成产物，手改容易被下次 build 覆盖。
- `dist/`
  - 这是构建输出目录，直接改没有意义。
- 全局桥接名
  - 如 `__AM_HOOK_MANAGER__`、`__ALIMAMA_OPTIMIZER_TOGGLE__`、`__AM_WXT_PLAN_API__`，这些名字有测试和运行时依赖。
